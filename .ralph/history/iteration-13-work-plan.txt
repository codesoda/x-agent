## Architecture Review

### Overall system design and component boundaries
`gha-agent` is a single-step domain wrapper around the existing shared runtime and orchestration conventions. The architecture is unchanged relative to existing agents.

- Script layer: `skills/gha-agent/scripts/gha-agent.sh` handles argument parsing, discovery, scoped command execution, and result rendering.
- Shared runtime: `lib/x-agent-common.sh` supplies common setup, locking, output formatting helpers, fail-fast semantics, and dependency checks.
- Registration layer: `skills/gha-agent/SKILL.md`, `install.sh`, and `README.md` register the agent and document usage.
- Validation layer: `tests/gha-agent/*` and `tests/run-scenarios.sh` validate fixture behavior and shell compliance.

Dependency flow is linear: plan/scope/validate -> discovery -> command execution -> output/logging -> aggregated overall result.

### Dependency graph and coupling concerns
`gha-agent` depends on:
- local path conventions via `SCRIPT_DIR/../../../lib`
- `actionlint` binary availability in PATH
- test fixtures under `.github/workflows`

`tests/run-scenarios.sh` depends on exact script path and executable bit. `install.sh` depends on the same paths for distribution and discovery.

Coupling risks:
- Path coupling: wrong `LIB_DIR` resolution breaks all shared helpers.
- Install coupling: missing `gha-agent` in `SKILLS` makes agent undiscoverable.
- Tool coupling: absence of `actionlint` should gracefully skip via fixture tool requirements, not script crash.

### Data flow patterns and potential bottlenecks
Data flow per invocation:
1. Parse flags/command and help path.
2. `need actionlint` then shared `setup_outdir` + `setup_lock`.
3. Resolve workspace scope from `CHANGED_FILES` or `.github/workflows` discovery.
4. Run actionlint command.
5. Write full output to `$OUTDIR/lint.log` and emit trimmed diagnostics.
6. Print final overall pass/fail and return exit code.

Bottlenecks are minimal and limited to filesystem walks for workflow file discovery and `actionlint` execution over YAML files.

### Scaling characteristics and single points of failure
- Scales linearly with number of workflow files and size of workflow YAML.
- Single-point-of-failure remains the lockfile and temp log lifecycle per agent name; if lock cannot be acquired because of stale FD or FS issue, runs serialize or proceed (depending on system behavior).
- In lockless environments, overlapping runs can contend for shared `.github/workflows` read operations but these are read-only; output interleaving risk is avoided by per-run log dir.

### Security architecture
- No external network/API interactions are introduced.
- Inputs are local paths/environment variables and should be treated as trusted because agents are run inside a local repo.
- Defensive controls:
  - command execution only through fixed executable `actionlint` with validated path arguments.
  - quoting of discovered file paths in argument arrays to reduce glob/word-splitting effects.
  - shared lock prevents concurrent overlapping runs from clobbering workspace assumptions.

### Key architectural options and tradeoffs
Decision A: CHANGED_FILES behavior
- Option 1: Ignore `CHANGED_FILES`; always run `actionlint` on all workflow files.
  - Effort: lowest, simplest.
  - Risk: slower and violates FR-3/scoped behavior.
  - Future impact: weakens incremental behavior as specs scale.
- Option 2: Filter CHANGED_FILES to workflow paths and pass scoped list; fall back to all when empty.
  - Effort: low.
  - Risk: scoped file expansion edge cases with spaces/unusual filenames.
  - Future impact: matches broader repo style and gives strong incremental behavior.
- Option 3: Implement a generic scope abstraction for all agents in shared lib.
  - Effort: high.
  - Risk: migration ripple to non-GitHub agents.
  - Future impact: possible reuse but out-of-scope for single spec.

Chosen: Option 2. It satisfies FR-3 and keeps future scalability while avoiding cross-spec churn.

Decision B: Scope discovery strategy
- Option 1: `find .github/workflows -type f` and filter extensions.
  - Effort: low.
  - Risk: extra scan if repo has many nested workflow-like paths.
  - Impact: predictable and easy to test.
- Option 2: Use `git ls-files ".github/workflows"` only.
  - Effort: low.
  - Risk: misses untracked workflow additions.
  - Impact: weak for dirty working directories.

Chosen: Option 1 (filesystem-first) for correctness in all working-tree states.

Decision C: Skip semantics when no workflows exist
- Option 1: Return FAIL with explicit instruction.
  - Impact: alerts caller but may be noisy in repos without workflows.
- Option 2: Return SKIP with explicit reason.
  - Impact: aligns FR-4 and consistent with other agents.

Chosen: Option 2.

Implications for future specs: keep this same skip-first pattern in new single-scope agents (helm/docker/kube/etc.) when discovery paths are optional.

## Code Quality Planning

### Code organization and module structure
Keep `gha-agent` domain file under ~200 lines by relying on shared helpers for:
- cleanup and locking
- step timing
- result formatting
- lock acquisition

`gha-agent.sh` should therefore contain only:
- header + shared-lib source
- usage/help text
- scope resolver
- one command function (`run_lint`)
- command dispatcher (`main`)

### DRY opportunities and repetition plan
Use shared `print_result`, `setup_outdir`, `setup_lock`, `step`, and `fmt_elapsed`; avoid duplicating trap/lock code. The only domain-specific duplication may be argument normalization for actionlint command modes.

### Error handling patterns for explicit edge cases
All edge cases must map to deterministic output and exit semantics:
- missing required argument context for help should not fail on `need()`
- missing `actionlint` should fail with code 2 via `need`
- no `.github/workflows` directory should produce `Result: SKIP (no .github/workflows/ directory found)`
- CHANGED_FILES set but no matching workflow paths should produce `Result: SKIP` (clear reason)
- actionlint failures should produce `Result: FAIL` and `Fix:` hint
- successful run should clean temp logs unless `KEEP_DIR=1`
- keep `overall_ok` and `--fail-fast` behavior consistent in command path

### Technical debt prevention
- Keep path filtering logic centralized in one helper function and reused by `run_lint`.
- Keep fix hint text standardized and deterministic.
- Keep failure output length bounded by `MAX_LINES`.
- Do not add per-agent custom trap logic that bypasses shared lib.
- Keep CHANGED_FILES parsing consistent with existing agents (space-delimited list assumption) to avoid creating new divergence.

### Engineering level
Use `engineered enough` behavior:
- no over-abstracted generic YAML parser
- explicit checks for expected files
- straightforward bash arrays for scoped files
- avoid brittle command substitutions with unguarded glob expansion.

## Test Strategy

### Coverage model
- Unit/isolated behavior is command-level in shell execution terms (scope resolution and conditional branch behavior).
- Integration behavior is `tests/run-scenarios.sh gha-agent` with clean and issues fixtures.
- End-to-end validation includes running full repository checks through `ralph validate work`.

### Critical edge-case coverage
- clean workflow: valid `ci.yml` passes with `EXPECT_EXIT=0`
- issues workflow: invalid workflow fails with `EXPECT_EXIT=1`
- missing workflow directory: runs should SKIP and return PASS
- CHANGED_FILES scoped to non-workflow files: should SKIP if no workflow matches
- CHANGED_FILES scoped to one workflow file: should only lint that file
- all mode on skip case: default still follows output contract
- `--fail-fast` with an explicit failing lint: no extra steps exist, but guard still exercised and should not regress
- install/README presence checks via review against touched files.

### Failure modes and error paths
- absent `actionlint` in environment
- corrupted workflow YAML causing actionlint parser errors
- scenario wrapper passing spaces in workflow filenames (documented limitation)
- help invocation with nonexistent project root/toolchain
- missing `.github/workflows` directory

### Assertion strength expectations
- assert exact exit codes through `EXPECT_EXIT`
- assert `Result:` lines include PASS/FAIL/SKIP states
- assert `Fix:` exists on FAIL
- assert logs are produced under `$OUTDIR` and `Overall` summary appears
- assert skip-path messages are explicit and actionable.

## Performance Considerations

- No database usage.
- Avoid repeated file-system scans by resolving scope once before running any checks.
- `find .github/workflows` is O(n) in repository size; in monorepos this is bounded and one pass.
- `actionlint` runtime is the dominant cost; CHANGED_FILES reduces this in incremental scenarios.
- Logging to disk is O(output size) and safer than memory buffering.
- No caching required for first implementation; cacheable future optimization: optionally store last-scan path list per run path if repeated invocations become hot.

## Decision Log

- Decision 1: Use per-spec shared-lib source pattern instead of local utility duplication.
  - Options considered: keep duplicated bootstrap code in new script, or source `lib/x-agent-common.sh`.
  - Chosen: source shared lib.
  - Why: consistent with backlog scope and line budget.
  - Impact: lower maintenance, same patterns as existing agents, uniform fail-safe semantics.

- Decision 2: Implement strict skip-first behavior for missing `.github/workflows`.
  - Options considered: return FAIL, return SKIP, auto-create dir.
  - Chosen: `SKIP` with explicit reason.
  - Why: aligns FR-4 requirement text and avoids false failures in non-GHA repos.
  - Impact: predictable behavior for repos with only Terraform/Python etc.

- Decision 3: Preserve local command semantics for `all` with one-step body.
  - Options considered: special-case `all` with no step expansion vs route through `lint`.
  - Chosen: route `all` through step dispatch to keep consistent `overall_ok` and fail-fast handling.
  - Why: future-proofing for additional steps.
  - Impact: minimal complexity and consistent UX across agents.

## Edge Cases and Risks

- `.github/workflows` directory does not exist in repo.
- CHANGED_FILES set but no workflow path matches and no direct discovery files exist.
- CHANGED_FILES contains non-workflow files only.
- CHANGED_FILES path separators differ on platform (existing convention keeps space-delimited paths only).
- `actionlint` unavailable even though fixture says required tool present.
- actionlint emits non-zero while no output, or output without diagnostics.
- Help invoked before toolchain install or from non-repo directory.
- CI versus local behavior around log line truncation due shared MAX_LINES defaults.
- Installer path mismatch on remote install if `gha-agent` files absent or script not executable.
- README/AGENTS snippets can become stale if not updated with new agent line.
