## Architecture Review

### System Design and Component Boundaries
The kube-agent is a command-wrapper service that follows the existing x-agent composition pattern: one domain script for orchestration, one SKILL contract, one scenario suite, and two repository metadata touchpoints (`install.sh`, `README.md`). Runtime responsibilities stay in `skills/kube-agent/scripts/kube-agent.sh`; shared runtime behavior remains in `lib/x-agent-common.sh`.

The domain script owns:
1) CLI parsing for `validate` and `all`,
2) external tool detection and selection,
3) manifest discovery and scope resolution,
4) one validation execution path with unified logging, and
5) output contract rendering.

Shared dependencies stay outside domain logic, including output helpers, temp dir lifecycle, lock handling, and fail-fast gating.

### Dependency Graph and Coupling
The execution critical path is local-first and file-driven. `kube-agent` depends on one of `kubeconform` or `kubeval`. It also depends on repository files and environment knobs. Coupling points:
1) Path conventions (`SCRIPT_DIR/../../../lib`, `OUTDIR`, `TMPDIR_ROOT`) inherited from existing agent scripts.
2) Scenario runner expectations (`skills/.../scripts/...sh`, scenario env variables, exit-code contract).
3) Installer and metadata coupling in `install.sh` and `README.md` for discoverability.
4) `lib/x-agent-common.sh` for shared behavior and clean process semantics.

### Data Flow and Failure Paths
Input enters via env vars (`CHANGED_FILES`, `RUN_VALIDATE`, `KUBE_SCHEMAS_DIR`, universal knobs) and command args. After `--help` handling and tool resolution, `kube-agent` resolves a deterministic manifest set, then runs one validation step.

Data flow details:
1) `main` parses help/flags/command.
2) `main` creates `OUTDIR` and (optionally) lock.
3) discoverer collects candidate YAML files.
4) scope filter applies `CHANGED_FILES` when present.
5) manifest filter applies `apiVersion:` and `kind:` checks.
6) validation command runs on collected files.
7) `print_result` and `print_overall` produce structured summary.

### Scaling Characteristics and Bottlenecks
Bottlenecks are filesystem traversal and external validator runtime. On large monorepos with many YAML files, scan + grep dominates. Large changed file lists can amplify scope resolution cost if not capped. Validator runtime scales approximately linearly with manifest count.

Single points of failure are: missing validator binaries, oversized file sets, and unstable tool output formats across versions.

### Security Architecture
No network calls or secret handling is required. Security risks are local command-invocation risks and path handling. Keep command execution argument-safe and avoid `eval`. Do not interpolate user-controlled flags directly into shell expansions unless quoted.

### Architectural Decision Options

Decision A: How to resolve manifest candidates.
Option 1: `find` recursively + extension filter + content grep for `apiVersion:` and `kind:`. Effort: low, low coupling, predictable. Risk: false positives in edge-case comments/strings, but bounded by domain expectation and test coverage. Maintenance: low, no extra dependencies.
Option 2: invoke `yq` or parser-based YAML introspection. Effort: medium, parser dependency required. Risk: extra dependency and portability issues across runner environments. Maintenance: medium.
Option 3: use Kubernetes client tooling directly for strict API object parsing. Effort: high, complexity and performance overhead. Risk: over-engineering versus project constraints.
Recommended: Option 1.

Decision B: How to invoke validators.
Option 1: one batched command with all discovered files. Effort: low, simpler logs, fewer processes. Risk: coarse per-file diagnostics. Maintenance: low.
Option 2: one command per file with immediate context. Effort: higher, larger number of subprocess calls. Risk: longer runtime on big repos. Maintenance: medium.
Option 3: hybrid mode by threshold count. Effort: medium/high. Risk: branching complexity and inconsistent output.
Recommended: Option 1.

Decision C: When to use `kubeval` branch.
Option 1: hard-require kubeconform only. Effort: low but lower compatibility. Risk: spec violation (kubeval must be accepted fallback).
Option 2: prefer kubeconform then fallback to kubeval. Effort: low, aligns with FR-6. Risk: output parsing differences across tools.
Option 3: try both and merge failures. Effort: medium/high, noisy output and longer runtime.
Recommended: Option 2.

Decision D: Scope testing policy.
Option 1: only clean/issues fixtures. Effort: low. Risk: silent regressions in `CHANGED_FILES` and skip behavior.
Option 2: add dedicated no-manifest and scoped fixtures. Effort: moderate. Risk: extra fixture maintenance. Benefit: catches branch-specific regressions early.
Recommended: Option 2.

### Implications for Future Specs
Keep discovery/filter helpers thin and explicit because kube-agent introduces a discover/filter/validate pattern likely to be reused by docker/ansible/sql/gha-like agents. Add fixture branches from this spec to reduce future FR-3 regressions.

## Code Quality Planning

### Module Structure and Organization
Keep `kube-agent.sh` small by isolating domain logic into separate helpers: discover, filter, validate, output. Shared behavior must stay in `lib/x-agent-common.sh`, including setup, output contract, and control primitives.

### DRY Opportunities and Anti-Drift
High-value dedup points are already available in common lib: `setup_outdir`, `print_result`, `print_overall`, `should_continue`, `need`. The domain script should only define one validator dispatcher function and one validation command builder to avoid duplicated command logic for kubeconform/kubeval.

### Explicit Error Handling Plan
Failure paths to implement explicitly:
1) Neither validator installed: print both options and exit `2`.
2) Unsupported command: usage message and exit `2`.
3) `kubeconform`/`kubeval` returns non-zero: print truncated output, `Fix` hint, log path, return fail status.
4) No YAML or no Kubernetes manifests after filtering: `Result: SKIP` and keep exit `0`.
5) Empty `CHANGED_FILES`: treat as no filter and run full discovery.
6) `CHANGED_FILES` paths pointing to non-files: ignore safely and continue.
7) Schema dir passed but invalid: pass-through and leave underlying tool to report.
8) File list too long for command line: prefer one invocation but document fallback path if needed.

### Technical Debt Controls
- No associative arrays or Bash 4-only features.
- Do not implement custom parsing where existing helper abstractions already exist.
- Keep fix-hint and skip text stable across changes.
- Avoid hidden behavior: every branch should print a human-readable reason.

### Engineering Level
Target implementation is lean and robust, no over-abstraction. The goal is deterministic behavior with explicit branches for optional scoping and fallback tool selection.

## Test Strategy

### Coverage Plan
Primary verification remains scenario-driven via `tests/run-scenarios.sh` because there is no dedicated unit test framework in this repo. Keep the validation signal as:
1) harness-level exit assertions,
2) `shellcheck --severity=warning`, and
3) manual targeted checks for output text on branch-sensitive behavior.

### Required Test Matrix
- `tests/run-scenarios.sh kube-agent` (full suite for kube-agent only).
- `tests/run-scenarios.sh` (global pass as a smoke check after edits).
- `shellcheck --severity=warning` via harness and direct invocation if needed.
- Scenario-level fixture expansion for scoped and skip branches.

### Edge Cases and Expected Assertions
- clean fixture with valid Deployment passes (`EXPECT_EXIT=0`, `REQUIRED_TOOLS=kubeconform`).
- issues fixture with invalid manifest fails (`EXPECT_EXIT=1`, `REQUIRED_TOOLS=kubeconform`).
- no manifest fixture yields `SKIP` and exits `0`.
- non-Kubernetes YAML fixture contains `apiVersion` but no `kind` (or vice versa) and must be skipped.
- CHANGED_FILES scoping where only Kubernetes manifests are inside set.
- CHANGED_FILES no-match (non-YAML or non-k8s YAML) yields `SKIP`.
- CHANGED_FILES includes deleted/nonexistent file entries, should still be robust.
- fallback tool path when kubeconform is absent and kubeval is present.
- no tool path: tool detection exits `2` with both install hints.
- `KUBE_SCHEMAS_DIR` custom path included in invoked command for both tools.
- `--help` works without validator installed.

### Failure Modes and Error Path Tests
- command-path unknown command exits `2`.
- no valid manifests found after discovery returns `SKIP` not `FAIL`.
- malformed manifest causing validator parse fail returns `1` and `Fix` hint.
- command dispatch path where `RUN_VALIDATE=0` and `all` should skip and still report pass if no other failures.

### Quality Expectations
Assertions should check not only exit code but also structured text when possible:
- `Validator: kubeconform` and `Validator: kubeval` output when branch chosen.
- `Result: PASS|FAIL|SKIP`, `Fix:` on failures.
- `Full log: <path>` path exists and is stable for `SKIP`/`FAIL`.
- `Overall: PASS|FAIL` appears once.

## Performance Considerations

### I/O and Traversal Costs
`find` plus `grep` is the dominant local cost. Pruning directory trees early (`.git`, `.github`, `.git` directories, `node_modules`, `charts`) is essential to avoid scanning generated/dependency trees.

### Command Invocation Strategy
Use one external command call for validation per run to avoid N+1 subprocess overhead. Keep argument list in an array-like helper style where supported; if the argument array becomes too large, we may need a fallback chunking strategy (only if needed in large repos).

### Memory and Log Footprint
Store discovered file list as newline-separated plain strings in variables to avoid large arrays and keep compatibility with Bash 3.2. Logs go to temp files in `OUTDIR` and are only previewed (first `MAX_LINES`) on terminal.

### Caching and Future Optimization
No cross-run cache is justified. Per-run cache of discovered file list and validator selection is sufficient.

### Hotspots and Risky Paths
- Large `find` in monorepos.
- Large manifest files with comments that trigger broad grep matches.
- Parsing summary output differences between validator versions.
- Frequent lock contention if lock is added and runs overlap.

## Decision Log

Decision 1: Prefer kubeconform first, fallback to kubeval.
Options: (a) hard-require kubeconform, (b) prefer kubeconform and fallback to kubeval, (c) attempt both and combine diagnostics.
Chosen: (b).
Impact: deterministic behavior, stable support matrix, minimal runtime, explicit compatibility.
Implications: future validators can be swapped into a single resolver branch if needed.

Decision 2: Use lightweight grep-based manifest classification.
Options: (a) extension-only with `apiVersion` + `kind` content filter, (b) YAML parser-based extraction, (c) Kubernetes-aware parse via additional tooling.
Chosen: (a).
Impact: zero new dependencies and fast implementation; explicit test coverage required for false positives.
Implications: include no-manifest and non-k8s YAML fixtures to keep risk bounded.

Decision 3: Enforce branch-specific scenario coverage in test plan.
Options: (a) rely only on clean/issues, (b) add scoped and skip branches as dedicated scenarios, (c) add scripted ad-hoc test commands outside harness only.
Chosen: (b).
Impact: better regression protection where full scans can hide CHANGED_FILES and skip-path bugs.
Implications: follow this rule for future scoped agents (helm/dock/ansible/sql/ etc.).

## Edge Cases and Risks

- CHANGED_FILES contains spaces; current environment parser is whitespace-delimited and not space-safe.
- `apiVersion:` / `kind:` keys in comments can create false positives.
- Validator output formats change across versions and may break count extraction.
- `kubeval` strict mode behavior differs from `kubeconform` summary model.
- Very large manifest sets may exceed shell argument limits; add fallback chunking only if needed in implementation.
- `setup_lock` optional for concurrency; decide if needed before finalizing.
- `charts/` exclusion may unintentionally skip legitimate manifests in chart repo roots; this is intentional to avoid helm domain overlap.
- No tool installed: `run-scenarios` may skip instead of fail, so manual environment checks remain required for full pass.
- Installer remote mode may need explicit path assumptions for future local copy of `SKILL.md` and script path.
- Documentation and install paths can drift from actual command names if not synchronized in one pass.
