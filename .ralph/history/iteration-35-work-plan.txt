## Architecture Review
### Overall System Design and Boundaries
- The new SQL runner is a thin domain-specific layer over the shared runtime (`x-agent-common.sh`) rather than a separate architecture.
- Boundaries:
  - Discovery layer (find + CHANGED_FILES filtering).
  - Execution layer (step handlers: lint/fix).
  - Control layer (argument parsing, modes, and output orchestration).
  - Integration layer (install + SKILL.md + fixtures + harness visibility).
- Coupling profile:
  - Tight local coupling to `sqlfluff` CLI flags (`lint`, `fix`, `--dialect`, `--force`).
  - Runtime coupling to shared helpers remains low and intentional (uniform style, lock, temp logs, printing semantics).
  - Fixture/test coupling should remain at harness level only.

### Dependency Graph and Coupling Concerns
- Core dependency path: `sql-agent.sh` -> `lib/x-agent-common.sh` -> shell utilities (`find`, `wc`, `head`) + optional `sqlfluff`.
- Install path dependency: `install.sh` must know about the new skill for downloads and optional dependency messaging.
- Documentation dependency: `README.md` must reference new skill and usage.
- Risk: if discovery logic duplicates patterns from other agents, future fixes may drift; contain by mirroring existing conventions and helper style.

### Data Flow and Potential Bottlenecks
- Data flow per invocation: env/config parse -> target collection -> per-command tool execution -> result logs/contract output -> cleanup.
- Bottlenecks:
  - Repository-wide `find` on very large trees (worst-case I/O and syscall heavy).
  - Invoking `sqlfluff` per fixture set across many files can dominate wall time.
  - `MAX_LINES=999999` in CI can emit large local output; still bounded by tool output file.
- Scaling characteristics:
  - O(F) file traversal; O(T) tool runtime where T tracks SQL statements and parser complexity.
  - Lock serializes concurrent runs per agent name, avoiding contention races but limiting parallelism intentionally.

### Single Points of Failure
- `sqlfluff` installation / version mismatch and unsupported dialect settings.
- `find` failures on permission-limited paths (mitigated by pruning known folders and local execution location).
- Incorrect handling of `CHANGED_FILES` (empty/relative paths) can produce false SKIP or over-inclusive linting.

### Security Architecture
- Authentication: none; local-only CLIs only.
- Data access: local filesystem read of discovered SQL files and write to local temp logs.
- Input boundaries: tool args are derived from environment + controlled paths; avoid unquoted expansions to prevent word splitting and path expansion bugs.
- Output boundaries: full logs are local; success path can delete logs unless `KEEP_DIR=1`.

### Architectural Decisions (with alternatives)
1) Discovery strategy for `.sql` files
- Option A (Recommended): shared `find` recursion for all modes with exclusions and direct `CHANGED_FILES` filtering.
  - Effort: low.
  - Risk: broad scans in very large repos, but predictable.
  - Impact: aligns with existing agents and minimizes divergence.
- Option B: rely only on `CHANGED_FILES` and fail when unset.
  - Effort: low.
  - Risk: regressions in CI where CHANGED_FILES is absent.
  - Impact: misses intended auto-discovery use case; fails acceptance.
- Option C: add language-aware index cache file.
  - Effort: high.
  - Risk: stale cache and complexity.
  - Impact: unnecessary for current scope.

2) Fix execution order in `all`
- Option A (Recommended): when fix enabled run `fix` first, then `lint`.
  - Effort: low.
  - Risk: requires careful mode handling to avoid running fix in CI.
  - Impact: meets spec and produces lint results on repaired code.
- Option B: keep existing command order as declared (`lint` first).
  - Effort: lowest.
  - Risk: lint on stale violating requirement.
  - Impact: cannot verify post-fix state in output.
- Option C: separate fix in a chained external mode only.
  - Effort: medium.
  - Risk: extra complexity for callers.
  - Impact: weaker `all` contract.

3) Process model for `sqlfluff` invocation
- Option A (Recommended): single invocation with all files in one call.
  - Effort: low.
  - Risk: large argument list if huge file set.
  - Impact: clearer logs and faster overall tool startup.
- Option B: one call per file.
  - Effort: medium.
  - Risk: noisy logs and slower due repeated startup.
  - Impact: simpler progress mapping, but less efficient.
- Option C: temporary list-file + xargs.
  - Effort: medium/high.
  - Risk: portability issues on Bash 3.2 and xargs handling.
  - Impact: fragile for current constraints.

### Implications for Future Specs
- Keep SQL implementation pattern aligned with go/terraform/bash style to reduce review friction.
- Explicitly add command coverage and scoped fixtures because prior iterations showed scoped/command gaps can be missed.
- Avoid adding ad-hoc custom helper libraries; continue extending shared lib only for high-reuse behavior.

## Code Quality Planning
### Module and Organization
- Keep sql-agent in one file under 200 lines of domain logic by delegating shared concerns to `x-agent-common.sh`.
- Use small named functions: `resolve_fmt_mode`, `discover_sql_files`, `run_lint`, `run_fix`, `main`.
- Keep fixture changes isolated to `tests/sql-agent/**` and docs/installer updates.

### DRY Opportunities and Reuse
- Reuse shared output/trap/lock/need helpers from `x-agent-common.sh`.
- Reuse discovery pattern shapes across existing agents: `CHANGED_FILES` filtering first, then full-project discovery.
- Use `print_result`, `step`, `fmt_elapsed`, and `print_overall` instead of custom output formatting.

### Error Handling Pattern (explicit)
- Invalid `FMT_MODE`: fail fast with `exit 2` and clear message.
- Missing `sqlfluff`: `need sqlfluff` -> `exit 2`.
- Empty target set:
  - No `.sql` files at all -> `Result: SKIP (no .sql files found)` and success exit.
  - `CHANGED_FILES` set but no matching `.sql` -> same skip branch.
- `sqlfluff lint` failing: emit `Result: FAIL` + exact fix hint, include first `MAX_LINES`.
- `sqlfluff fix` failing under `RUN_FIX`/`FMT_MODE=fix`: fail with explicit manual-fix hint, still continue according to `FAIL_FAST`.
- CI forcing check mode: if `FMT_MODE=fix` in CI, coerce to `check` unless command is explicit `fix` invocation.
- Help command path: print usage before tool checks to avoid false positives in empty projects.

### Technical Debt Prevention
- Do not hardcode log filenames beyond expected naming; use `<step>.log` under `OUTDIR`.
- Keep dialect default centralized in variable default (`ansi`) and documented in SKILL/README.
- Use quoted arrays for command input when expanding file list to avoid word-splitting regressions.
- Keep behavior consistent with exit codes (`0` pass, `1` fail, `2` usage/config).

### Engineering Level
- Implement robust-but-limited scope: enough checks to satisfy required behaviors and FRs without over-abstracting.
- Avoid adding caching layers or JSON parsing dependencies; not warranted for one command.
- Use plain `find` + loop + `printf`/`wc` like peer agents for consistency and auditability.

## Test Strategy
### Coverage Plan
- Unit-level (script behavior by direct execution):
  - Dialect override, FMT_MODE resolution, CHANGED_FILES filtering, and all-command ordering.
- Scenario/e2e-level (`tests/run-scenarios.sh`):
  - `clean/` and `issues/` base pass/fail paths.
  - Scoped discovery and no-match behavior.
  - `fix` command explicitly.
  - `CI=true` check-mode enforcement.
  - no-sql skip path.
  - dialect-specific pass-through path validation.

### Critical Edge Cases
- No `.sql` files in repo.
- CHANGED_FILES includes non-existent paths.
- CHANGED_FILES includes non-sql files.
- CHANGED_FILES includes only nested/relative sql files with `./` prefixes.
- Mixed file tree with directories to prune (`.git`, `node_modules`, `vendor`).
- `REQUIRED_TOOLS` missing: scenario should be skipped.
- `RUN_FIX=1` in CI should not mutate files (check-only run remains).
- `FMT_MODE=fix` with no issues should pass and not modify clean fixtures unexpectedly.
- `sqlfluff` unavailable: hard fail due required dependency.

### Failure Modes and Error Paths
- `find` returns nothing unexpectedly.
- `sqlfluff` returns non-zero with empty log output.
- Incorrect dialect value causing immediate command failure.
- Fix step fails on unsupported syntax while lint also reports the same issue.
- `all` runs `fix` then `lint`: lint can still fail; overall should reflect final check pass/fail.

### Assertion Strength and Verification
- Use `EXPECT_EXIT` assertions in each fixture.
- Use fixture naming and environment exports in scenario files to assert targeted branches.
- Use explicit command variants in RUN_ARGS for command-level behavior (not only `all`).
- Confirm logs are present and meaningful (quick manual checks for skip and command hints).
- Ensure `run-scenarios` plus shellcheck both pass before work handoff.

### Proposed Scenarios
1. `tests/sql-agent/clean`: valid SQL fixture (provided by spec), should exit 0.
2. `tests/sql-agent/issues`: lint-violating SQL fixture, should exit 1.
3. `tests/sql-agent/clean-dialect`: no-changes query only valid in mysql and exported `SQLFLUFF_DIALECT=mysql`, expect 0.
4. `tests/sql-agent/scoped-match`: export `CHANGED_FILES` to the issue SQL file, run `all`, expect deterministic branch and behavior.
5. `tests/sql-agent/scoped-no-match`: export `CHANGED_FILES` to non-sql files, expect skip/0.
6. `tests/sql-agent/no-sql`: no `.sql` files at all, expect skip/0.
7. `tests/sql-agent/fix-command`: run `fix` with issue fixture and `EXPECT_EXIT` aligned to fix success or failure path.
8. `tests/sql-agent/ci-check`: set `CI=true` and `RUN_FIX=1` on issue fixture with `RUN_ARGS="all"`, expect fail-only behavior consistent with lint-only mode.

## Performance Considerations
- Avoid per-file subprocess overhead by invoking `sqlfluff` once with aggregated file list; preserve logs for diagnostics.
- Use pruning in `find` (`.git`, `node_modules`, `vendor`) to reduce walk size.
- Keep all-string transformation and counts via `wc`/`printf` O(n) with small memory footprint.
- Avoid retaining full diagnostics in memory; read diagnostics from temporary log and stream with `head` for preview.
- No persistent cache needed; run-local helper should stay stateless per invocation.

## Test Plan for Performance and Regression
- Compare runtime between full auto-discovery and CHANGED_FILES-only fixture mentally via elapsed output to detect pathological full-tree scans.
- Watch for `fix` command invoking `sqlfluff` on huge project sets in CI and document explicit opt-in behavior (`RUN_FIX=0` default).
- Validate that `all` with auto-fix remains bounded by one lint + optional one fix pass.

## Decision Log
- Decision: Source shared runtime from `lib/x-agent-common.sh` in `sql-agent.sh`.
  - Options Considered: custom local helpers vs shared lib.
  - Chosen Approach: shared lib only.
  - Implications: Faster iteration, consistent output, smaller script, lower style drift.

- Decision: Keep all file discovery local with `find` and fixed prune list.
  - Options Considered: always rely on CHANGED_FILES vs full scan vs cache.
  - Chosen Approach: full scan fallback with deterministic pruning.
  - Implications: predictable behavior and parity with existing agents.

- Decision: Require dialect and command-specific fixtures before implementation complete.
  - Options Considered: only clean/issues fixtures vs full command/scoped matrix.
  - Chosen Approach: add explicit scoped, no-sql, fix, CI fixtures.
  - Implications: reduces regressions seen in earlier scoped agents; slightly larger test matrix.

## Edge Cases and Risks
- Empty repos or repo without `.sql` files.
- `CHANGED_FILES` points to deleted or non-`.sql` paths.
- CHANGED_FILES path format mismatches (leading `./`).
- `sqlfluff` supports dialect-specific syntax differences; incorrect defaults can create false negatives/positives.
- CI forcing check mode while users expect fix: explicit command `fix` should remain opt-in but `all` must stay lint-only.
- `sqlfluff fix` can only safely rewrite some constructs; fixture should include manual-fix hint path.
- Concurrent scenario execution can overlap locks if lock filename path incorrect.
- Overly permissive discovery could include vendored/generated SQL in `node_modules`/`vendor` unless pruned.
- Failing to keep output contract breaks downstream automated consumption and review checks.
