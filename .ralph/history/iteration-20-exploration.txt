## Patterns & Conventions
- All scripts are Bash with `set -euo pipefail` and helper-driven output framing.
- `--help`/`help`/`-h` must be parsed before tool checks and project assumptions.
- `need` for hard dependencies exits with code 2 and user-facing message.
- `setup_outdir` and `print_result` are used for standardized log handling and contract compliance.
- `MAX_LINES` behavior defaults to 40 locally and 999999 in CI.
- `RUN_<STEP>`, `KEEP_DIR`, `MAX_LINES`, `FAIL_FAST`, `CHANGED_FILES`, `TMPDIR_ROOT` are universal knobs.
- `SKILL.md` `allowed-tools` includes every environment knob and the agent command path.
- `RESULT: FAIL` lines must include `Fix:` hints; SKIP is allowed for missing optional dependencies or no-op scope.
- `tests/run-scenarios.sh` skips scenarios when a required tool is missing and expects `EXPECT_EXIT` for non-skipped runs.

## Database Schema
- No SQL/NoSQL schema exists for runtime agent operation.
- Metadata used by RALPH is file-based (`.ralph/` state/log files) and not part of the runtime checks here.

## Test Infrastructure
- Test entrypoint: `tests/run-scenarios.sh`.
- Shellcheck runs on all scripts in `skills/`, `lib/`, and root/test scripts before fixtures by default.
- Fixture contract in `scenario.env`: `SCENARIO_NAME`, `AGENT_SCRIPT`, `RUN_ARGS`, `EXPECT_EXIT`, optional `REQUIRED_TOOLS`.
- `CHANGED_FILES` can be injected from `scenario.env` for scoped execution validation.
- `tests/run-scenarios.sh` compares fixture output only by exit code and prints logs for failures.
- Additional fixtures are recommended where FRs require scoped discovery and skip paths, especially for `CHANGED_FILES`.

## Architecture & Data Flow
- Input: environment + args -> argument parser -> dependency resolution -> discovery -> command execution -> logged output + structured step status -> overall result.
- Shared library functions are imported by each agent script and provide deterministic output/cleanup contract.
- kube-agent architecture must remain isolated: no mutable shared artifacts beyond temp logs and existing working directory reads.
- Validation commands are external to repo and called per discovered manifest set with result aggregation in script state.
- Outputs go through shared formatting and are also persisted in `OUTDIR` for detailed failure inspection.
