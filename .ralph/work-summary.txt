## Deviations from work plan

- **No `setup_lock` call**: bash-agent does not use workflow locking. Unlike cargo-agent (which has heavy build operations), `bash -n` and `shellcheck` are lightweight read-only checks with no shared state. This matches npm-agent and terra-agent which also don't use locks.
- **Issue fixture uses SC2034/SC2154 (warning-level)** instead of SC2086 (info-level): SC2086 is severity `info`, not `warning`, so it doesn't trigger at the default `--severity=warning`. Used unused-variable (SC2034) and unset-variable (SC2154) which are genuine warnings.
- **Modified `tests/run-scenarios.sh`**: Added `-not -path '*/issues/*'` to the shellcheck find command to exclude issue fixture scripts from the project-wide shellcheck scan. Without this, intentionally-bad fixtures in `tests/bash-agent/issues/` would fail the global shellcheck step.

## Edge cases handled
- `CHANGED_FILES` filtering: only existing `.sh` files are included
- No `.sh` files discovered: prints `Result: SKIP (no matching .sh files)` and returns 0
- Help/--help/-h: handled before `need` checks so it works without shellcheck installed
- Unknown commands: print usage and exit 2
- Shellcheck wiki links: extracts unique SC codes from output, limits to first 5, builds wiki URLs
- `SHELLCHECK_SEVERITY` override: passes through to shellcheck `--severity` flag
- `FAIL_FAST`: uses `should_continue()` from shared lib in `all` command path

## Tests written/run
- `tests/bash-agent/clean/` — passes with exit 0 (valid script, all checks pass)
- `tests/bash-agent/issues/` — fails with exit 1 (shellcheck warnings in bad.sh)
- `shellcheck --severity=warning` passes on bash-agent.sh itself
- Full suite: all 14 scenarios pass (including all existing agents)

## Known limitations
- `bash -n` only checks syntax, not runtime errors — this is by design
- File discovery splits `CHANGED_FILES` on whitespace, so paths with spaces are not supported (matches convention used by all other agents)
- `shellcheck` is invoked once with all files as arguments; very large file lists could hit argument length limits (unlikely in practice)
