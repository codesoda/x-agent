## Architecture Review

### Component Boundaries
`kube-agent.sh` is a thin command layer over external validators. It owns CLI parsing, discovery, scope handling, and result formatting. `lib/x-agent-common.sh` supplies cross-agent primitives for output, defaults, cleanup, and locking. Tests, installer metadata, and documentation are separate cross-cutting layers with no circular dependency.

### Dependency Graph and Coupling
Runtime dependencies are limited to one of `kubeconform` or `kubeval`; all other behavior is local Bash. The installer and README depend on deterministic skill naming. Scenario tests depend only on agent output contract and fixture layout, not on shared validation internals.

### Data Flow
Input enters via environment variables and CLI args. Discovery resolves candidate files from current working directory or `CHANGED_FILES` projection. Validation runs against those candidates using the selected tool and emits per-step result artifacts into `OUTDIR` plus structured terminal status. State flow remains small and linear: parse → discover → validate → report.

### Scaling Characteristics
Primary load is filesystem traversal. For large repos, recursive `find` dominates runtime and can be expensive if not pruned. Command invocation is linear in discovered file count. Validator runtime scales with file count and manifest complexity and is external-tool bound.

### Single Points of Failure
Validator availability is the clearest hard stop (`kubeconform` or `kubeval` absent). Discovery correctness is second-order: inaccurate exclusions can cause false pass/skip. `FULL LOG` preservation and deterministic exit codes are the recovery path for unknown behavior.

### Security Architecture
The script performs local file reads only. No network calls are required in baseline mode. Risks are mostly path-handling and command injection; avoid `eval` and keep command arguments in arrays when possible. Credentials are not expected and should not be introduced in logs.

### Architectural Decisions
Decision A: Manifest detection by extension plus grep patterns versus parser-based YAML parsing.
- Option 1 uses `find` and `grep` for `apiVersion` and `kind`.
  - Effort: Low. Risk: false negatives on nonstandard formatting.
  - Impact: fast to implement, aligns with existing project behavior.
  - Maintenance: low, minimal dependencies.
- Option 2 uses `yq` or a structural parser.
  - Effort: Medium. Risk: dependency on extra tooling and portability issues.
  - Impact: more semantic accuracy but stronger environment requirements.
- Option 3 parses with Kubernetes client libs through containerized tooling.
  - Effort: High. Risk: orchestration complexity and CI brittleness.
  - Impact: heavy and over-scoped for this script.
- Recommendation: Option 1 for now (simple, local, no extra toolchain), with strict tests for skip behavior and manifest filtering to control risk.

Decision B: Per-file validator invocation versus batched invocation.
- Option 1 runs one command on all discovered files at once.
  - Effort: Low. Risk: larger command lines and coarser error mapping.
  - Impact: faster in many repos and simpler logs.
- Option 2 runs validator per file in a loop.
  - Effort: Medium. Risk: larger log surfaces and slower run.
  - Impact: clearer file-level attribution.
- Option 3 hybrid approach with per-step batching and per-file fallback.
  - Effort: Medium/High. Risk: code complexity and inconsistent output format.
- Recommendation: Option 1, plus `KUBE_VALIDATOR` command wrapper and explicit result aggregation. It aligns with FR-8 style (one validation command line) and keeps logs compact.

Decision C: Scope strategy for kubeconform and kubeval.
- Option 1 uses unified wrapper function with command/flag branching.
  - Effort: Low. Risk: duplicated output handling if diverging flags.
  - Impact: clean implementation and predictable behavior.
- Option 2 writes separate functions per validator.
  - Effort: Medium. Risk: code duplication and drift.
  - Impact: easier independent debugging but harder to keep parity.
- Option 3 delegates to a shared output adapter with map-based dispatch.
  - Effort: Medium. Risk: abstraction overhead for one agent only.
  - Impact: reusable but unnecessary complexity now.
- Recommendation: Option 1 with shared collection helpers and validator-specific execution branch; consistent with the ~200-line domain rule.

### Future Implications
The same discovery + validation wrapper shape can be re-used for future policy-oriented agents and should remain in a predictable template form: discover first, then dispatch by selected backend, then common result formatting.

## Code Quality Planning

### Organization and Structure
Keep `kube-agent.sh` to three clear sections: configuration, discovery, and execution. Keep shared behavior in the common lib only. Keep SKILL and fixtures isolated to `skills/kube-agent/` and `tests/kube-agent/` respectively.

### DRY Opportunities
Reusable helpers exist in common lib (`setup_outdir`, `print_result`, `print_overall`, `should_continue`, `need`). Use them consistently to avoid custom output code. Keep one function for manifest collection and one for command builder so kubeconform/kubeval differences stay small.

### Explicit Error Handling
Edge handling must include:
- Missing validators: print both install options, exit 2.
- Missing `AGENT_SCRIPT` dependencies in scenario tests: harness handles, not script.
- Unknown command: usage plus exit 2.
- Help before dependency checks.
- `CHANGED_FILES` empty or set to files outside target patterns: clean SKIP branch.
- No manifests after filtering: `SKIP` with explicit reason.
- `kubeconform` summary parse failures: degrade to generic output and mark FAIL with full log path.
- Directory traversal exceptions from `find`: ignore missing directories and continue gracefully.

### Failure Modes Pattern
Every failure path returns either `1` (validation failed) or `2` (bad config/tooling). `SKIP` remains `0` when explicit no-work path is expected.

### Technical Debt Controls
Avoid clever tokenization. Keep path matching to spaces-sensitive behavior consistent with existing agents. Do not introduce associative arrays or Bash 4+ features. Add explicit comments for non-obvious behavior (`charts/` exclusion and validator fallback order).

### Engineering Level
The implementation should be "engineered enough": one robust discovery function, one command execution function branch, and no premature framework extraction. Avoid overbuilding schema downloads or AST parsing not required by FRs.

## Test Strategy

### Coverage Plan
Unit-style tests are not currently present in this repository; use scenario tests as the primary verification layer and targeted command checks as supplement.
Implementation verification layers:
1. `tests/run-scenarios.sh` shellcheck gate.
2. `tests/run-scenarios.sh kube-agent` clean and issues fixtures.
3. Additional scoped/no-manifest fixtures to exercise FR-3 and skip branches.
4. Manual validator-path verification with and without `KUBE_SCHEMAS_DIR` to confirm passthrough.

### Edge Cases to Test
No manifests at repo root.
Only non-Kubernetes YAML files.
`CHANGED_FILES` set to YAML but outside Kubernetes set.
`CHANGED_FILES` set to valid Kubernetes manifest.
`KUBE_SCHEMAS_DIR` pointing to custom location.
Both tool-path branches: kubeconform first, kubeval fallback.
Directory exclusions for `.github/`, `.git/`, `node_modules/`, `charts/`.
`--help` before any dependency checks.
Command `validate` and default `all`.

### Failure Paths to Assert
Exit 2 when neither validator is found.
Exit 1 for invalid manifests in issues fixture.
Exit 0 for clean manifests and for no-manifest SKIP scenario.
`Result: SKIP` messages should remain user-facing and not hidden.
`Fix:` hint present on FAIL and references `/kube-agent validate`.

### Test Quality Standards
Assertions should be stronger than exit code alone where possible. Where harness is exit-only, add explicit log-content checks during manual validation, especially for:
- `Validator: kubeconform` message under tool selection.
- `Result: SKIP` reason clarity.
- `Full log` path printed.
- Count output exists on kubeconform path.
- Error pass-through from kubeval path when run.

## Performance Considerations

### N+1 and Scan Patterns
Avoid nested file-level validation calls that multiply command overhead. Prefer one manifest list and one validator call unless required per-tool behavior dictates otherwise. Prune `find` with `-path` exclusions at source to avoid scanning `node_modules` and `.git`.

### Memory Usage
Avoid building huge bash arrays with every path when possible; use newline-separated strings and per-step streaming to logs. This is adequate for large trees and keeps shell memory stable.

### Caching and Reuse
No cross-run cache is needed. Within a run, cache selected validator name and discovered manifest list to avoid recomputation between command paths.

### Hot Paths
Fast path is clean repos with no Kubernetes files: ensure low-cost discovery and immediate SKIP.
Slow path is large monorepos with many YAML files: tune exclusion list and avoid repeated grep passes if possible.

### Optimization Boundaries
Keep optimization bounded to discovery and external command invocation. Do not introduce semantic YAML parsing or API calls that would exceed the domain of this script.

## Decision Log

Decision 1: Use kubeconform-first fallback to kubeval.
Options: (a) require only one explicit tool, (b) prefer kubeconform then fallback to kubeval, (c) run both and accept on intersection.
Chosen: (b) based on explicit FR-6 and implementation simplicity.
Implications: keeps behavior deterministic for users and future agents with similar "primary optional fallback" semantics.

Decision 2: Use extension+content grep filtering for Kubernetes manifests.
Options: (a) cheap grep-based filtering, (b) YAML parse AST filtering, (c) Kubernetes CLI schema validation pass only.
Chosen: (a) for speed and zero extra dependency footprint, but with scoped-path tests to constrain drift.
Implications: avoids external parser dependency and keeps the script below the 200-line domain target.

Decision 3: Add dedicated CHANGED_FILES and no-manifest fixtures for kube-agent in addition to clean/issues.
Options: (a) spec-only clean/issues, (b) include two focused scoped/skip fixtures, (c) add broad fuzz-like fixtures.
Chosen: (b) because it directly enforces FR-3 and resolves known review feedback pattern on path scoping.
Implications: increases confidence and should be copied to remaining scoped agents in later specs.

## Edge Cases and Risks

Scope filtering misses expected files because of spaces in filenames.
Manifest discovery false positives from comments containing `apiVersion` and `kind`.
Validator output format differences across installed versions of kubeconform and kubeval.
CHANGED_FILES includes deleted paths or directories.
Schema location option passed with incorrect type or missing directory.
No manifests found while CHANGED_FILES set to YAML-like non-Kubernetes files.
`--help` invoked in minimal environment where helper scripts are missing.
Fixture-based test drift when external tool version changes behavior.
README/install divergence from script command names.

## Risks and Dependencies

Dependencies:
`kubeconform` or `kubeval` availability in the environment for positive-path scenarios.
Shared library stability in `lib/x-agent-common.sh`.
Scenario harness consistency for environment-variable injection.

Assumptions that may not hold:
`kubeconform` output includes machine-parseable summary fields used for counts.
External validator messages remain non-empty and useful on failure.
Workspace contains manifests under default ignore rules.
Repo path and fixture paths are accessible by relative invocation from fixture directories.
