## Architecture Review

### Overall design and boundaries
The implementation is a single new command-line agent bounded to `skills/bash-agent/scripts/bash-agent.sh`, metadata in `skills/bash-agent/SKILL.md`, scenario fixtures in `tests/bash-agent`, installer wiring in `install.sh`, and user documentation in `README.md`. The data flow is deterministic and linear:
- CLI/env parsing.
- Output directory + lock setup.
- File discovery.
- Step execution (`syntax`, `lint`, or `all`).
- Step logging + result formatting.
- Final `Overall` summary and exit code.

### Dependency graph and coupling
- Runtime dependencies for execution: `bash`, `shellcheck` (required), plus `find`, `grep`, `head`, `sort`, `uniq` from POSIX shell environment.
- Shared coupling: all common behavior comes from `lib/x-agent-common.sh` through path-based source import, so updates there affect all agents.
- Repo coupling: this spec touches five files and one optional metadata file, and is discovered automatically by `tests/run-scenarios.sh` by virtue of `tests/<agent>/scenario.env`.
- Installer coupling: adding `bash-agent` to `SKILLS` and `check_optional_deps` controls install-time messaging only; runtime behavior is independent once installed.

### Data flow and bottlenecks
- Input: user env + command + working directory.
- Transform: discovery list -> per-step command execution -> log file output -> parsed status/fix hints.
- Output: console summary + files under `$OUTDIR`.
- Bottleneck: scanning filesystem for `.sh` files recursively when `CHANGED_FILES` is empty; mitigated with excludes and short-circuit on empty discovery.

### Scaling characteristics and single points of failure
- Scaling is mostly linear in number of script files and shellcheck complexity.
- Single failure points are external tool execution (`bash`, `shellcheck`), lock contention, and directory traversal in large repos.
- Lock contention is serialized intentionally and can create queued latency but prevents state corruption.

### Security architecture
- Command execution is limited to the local project files and fixed tool binaries.
- Security risks are low relative to domain; primary risks are log path tampering and shell metacharacter issues in paths.
- Mitigations:
  - Quote expanded file names in tool invocations.
  - Filter `CHANGED_FILES` to file paths and existence checks.
  - Keep lock and temp-dir usage unchanged from shared library pattern.

## Architecture options (with tradeoffs)

### Decision A: shared-lib-first implementation
- Option 1: Keep all boilerplate local as in older agents.
  - Effort: low.
  - Risk: high long-term drift from shared style.
  - Impact: inconsistent behavior and larger per-agent scripts.
- Option 2: Use `x-agent-common.sh` helpers (`setup_outdir`, `setup_lock`, etc.) for all shared concerns and only implement domain logic.
  - Effort: moderate implementation effort, low maintenance.
  - Risk: stronger dependency on helper interface.
  - Impact: matches existing architecture and all future agents.
- Option 3: Wrap bash-agent in a wrapper script around a generic shellcheck checker utility.
  - Effort: medium+ (new infra).
  - Risk: abstraction overhead and extra indirection.
  - Impact: future extensibility but unnecessary complexity.
- Recommended: Option 2 for consistency and minimal future maintenance.

### Decision B: lint code extraction and fix-hint strategy
- Option 1: Static one-code hint only (first `SC####`).
  - Effort: very low.
  - Risk: under-informs when multiple shellcheck violations exist.
  - Impact: faster implementation, weaker fix signal.
- Option 2: Aggregate all unique `SC####` codes and append one wiki link per code.
  - Effort: moderate.
  - Risk: long `Fix:` line in noisy files.
  - Impact: highest signal for developers.
- Option 3: Print only tool output and no wiki links.
  - Effort: low.
  - Risk: violates FR-13 acceptance requirement.
  - Impact: insufficient for this spec.
- Recommended: Option 2, with link truncation to first 5 unique codes if necessary.

### Decision C: behavior when `CHANGED_FILES` resolves to zero files
- Option 1: Treat as PASS.
  - Effort: minimal.
  - Risk: may mask mis-scoped inputs.
- Option 2: Treat as SKIP with explicit reason.
  - Effort: low.
  - Risk: requires test expectations for SKIP path.
  - Impact: clear caller feedback and safer with CI automation.
- Option 3: Treat as FAIL.
  - Effort: low.
  - Risk: disruptive when hook calls intentionally scope empty sets.
  - Impact: highest risk for orchestration.
- Recommended: Option 2 (`Result: SKIP (no matching .sh files)`) to make state explicit without failing workflows.

## Code Quality Planning

- Keep script domain size small and readable by centralizing shared behavior in `x-agent-common.sh`.
- Use small helper functions: `collect_sh_files`, `run_syntax`, `run_lint`, and `extract_shellcheck_codes`.
- Follow existing naming style: `run_<step>()` and `usage()`.
- Preserve structured output contract exactly: `Step`, `Result`, optional `Fix`, `Output`, `Full log`, `Time`, `Overall`, and `Logs` from trap.
- Use one source-of-truth list (`sh_files`) to avoid duplicated file-selection logic across syntax and lint.
- Keep helper interactions explicit; avoid nested and hidden defaults.
- Avoid clever tricks; prefer straightforward loops and explicit status flags.
- Add comment lines near `CHANGED_FILES` and `SHELLCHECK_SEVERITY` for maintainability and future agents.

### Edge-case handling requirements
- CHANGED_FILES set and empty: validate each token is an existing `.sh` file.
- CHANGED_FILES contains missing files: ignore with warning summary count mismatch.
- Working directory has zero `.sh`: emit `SKIP` and exit 0.
- Syntax command returns non-zero on one file: continue scanning all files to collect all syntax failures in one pass.
- shellcheck output may include duplicate codes: deduplicate before composing Fix links.
- `SHELLCHECK_SEVERITY=error` should downgrade warnings and still run severity-specific checks.
- Unknown command and `-h|--help|help` should be handled before `need` checks that would block usage in non-project contexts.

## Test Strategy

### Coverage plan
- Unit-level command tests via direct invocation of `bash-agent` in fixture directories.
- Integration-level scenario tests through `tests/run-scenarios.sh bash-agent`.
- Static quality checks via shellcheck on the new script.
- Manual behavior validation in mixed shell-script repo to verify CHANGED_FILES scoping.

### Explicit test coverage matrix
- `help` command: prints usage and exits 0.
- `syntax` with clean fixtures: returns PASS and no `Fix:`.
- `syntax` with intentional syntax failure: returns FAIL, full log path present, `Fix:` includes re-run guidance.
- `lint` with clean scripts: returns PASS.
- `lint` with SC2086 fixture: returns FAIL, `Fix:` includes `https://www.shellcheck.net/wiki/SC2086`.
- `all` command: runs syntax then lint with `RUN_*` gating and `FAIL_FAST` support.
- `RUN_LINT=0` and `RUN_SYNTAX=0`: skip corresponding steps.
- `CHANGED_FILES` scoping: includes only matching `.sh` and excludes directories and non-`.sh` entries.
- `SHELLCHECK_SEVERITY=error`: ignore warning-only issues in fixture; pass/fail semantics follow chosen severity.

### Failure modes and error paths to verify
- Missing required dependency (`bash` or `shellcheck`) exits 2 with `Missing required tool: <tool>`.
- No scripts discovered after scan: `SKIP` and exit 0.
- lock already held: user sees wait message and command proceeds when lock is released.
- Scenario marked EXPECT_EXIT=1: exit 1 when syntax/lint fails.

### Test quality assertions
- Assertions should include both command exit codes and output semantics (`Result: PASS/FAIL/SKIP`, `Fix:`, `Logs:` path).
- For lint failures, assert wiki link format exactly `https://www.shellcheck.net/wiki/SC####`.
- Keep assertions deterministic by controlling `MAX_LINES` in tests when comparing output.

## Performance Considerations

- `find` is O(number of files); prune known directories to prevent scanning dependency trees.
- `bash -n` and `shellcheck` are per-file/external-process heavy; avoid unnecessary extra loops.
- Use one discovery pass so both steps share the list.
- Use log files for full output to avoid large in-memory buffers.
- Memory usage remains bounded because only shell command lines and small arrays are held in process memory.
- N+1 command patterns are avoided: shellcheck is invoked once with file arguments instead of per-file calls.

Decision: Skip step with explicit message.

- Treat as PASS.
- Treat as SKIP.

## Edge Cases and Risks

- `CHANGED_FILES` includes files with spaces in names: current split-based parsing is brittle; caller convention assumes space-separated file paths.
- `.sh` files outside repo root returned by CHANGED_FILES: filter to `-f` avoids attempting missing paths.
- Very large repositories with many `.sh` files: scan can be expensive; prune list should stay curated.
- External tool missing in minimal environments: hard dependency check must fail fast.
- `SHELLCHECK_SEVERITY` unknown: shellcheck should fail; script should surface command failure cleanly.
- Scenarios with only syntax errors and no shellcheck violations still need deterministic fail reason and logs.
- Windows line endings in fixtures: avoid regex assumptions in shellcheck code extraction that depend on Unix LF; use tolerant parsing.
