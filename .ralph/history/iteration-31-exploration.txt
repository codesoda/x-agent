## Patterns & Conventions
- Scripts are Bash 3.2-compatible and start with `set -euo pipefail`.
- `--help` / `help` / `-h` is handled before dependency or project checks so usage works without workspace context.
- Required dependencies use `need` and should exit with status 2 when missing.
- Output contract is strict and step-oriented (`Step`, `Result`, optional `Fix`, `Time`, `Overall`, `Logs`).
- `CHANGED_FILES` is optional; when set, discovery is narrowed and may become empty (then step-level SKIP expected).
- `MAX_LINES` is local=40, CI=`999999` unless overridden.
- Optional tools are surfaced as SKIP with explanatory reason, not hard fail.
- `shellcheck --severity=warning` runs on `skills/*`, `lib/*`, and test scripts before fixture execution.
- Scenario env values consumed by agents must be exported in `scenario.env` because the harness executes agents as a subprocess.
- For offline/network-sensitive validation, tests often pin deterministic behavior via explicit env knobs rather than external schema fetches.
- `KUBE_IGNORE_MISSING_SCHEMAS=1` was introduced for kube-agent to avoid network-coupled validation failures.

## Database Schema
- No application DB is used by the agent runtime.
- Control-state and workflow tracking live in `.ralph` metadata files (`ralph-state.json`, logs, and task/plan artifacts).

## Test Infrastructure
- `tests/run-scenarios.sh` discovers scenarios by scanning `tests/*/scenario.env`.
- Supports scoped filtering via `CHANGED_FILES` and dependency gating via `REQUIRED_TOOLS`.
- Exit expectations are numeric (`0` pass, `1` fail, `2` config/tool-missing failure).
- `shellcheck --severity=warning` failure aborts the scenario run before agent execution.
- Existing pattern: every scoped discovery behavior should be covered by dedicated fixtures (e.g., no-match/empty-match), especially for file-set filtering.

## Architecture & Data Flow
- `run-scenarios.sh` enters each fixture directory, sources `scenario.env`, and executes the configured agent script.
- Agent flow: parse CLI + help/version, initialize common globals, acquire lock (`setup_lock`), create `OUTDIR` (`setup_outdir`), discover candidates, run step functions, and emit step results.
- Per-step logs are written to files in `OUTDIR`; final output prints `Overall` and `Logs`.
- External dependencies are only local CLIs (`hadolint`, optional `docker`) and shell utilities (`find`, `grep`, `wc`, etc.).
- Installer path: `install.sh` must include new skill in `SKILLS`, optional dependency messaging, and AGENTS/CLAUDE snippet entry.

## Security / Boundaries
- No network calls are required by docker-agent core flow unless optional `docker build --check` is enabled and docker daemon is used.
- Command arguments are passed as local CLI/env values; avoid interpolating untrusted strings without quoting to prevent shell globbing/word-splitting issues.
- Tool execution is local-only; output logs are temporary unless `KEEP_DIR=1` or failure.
