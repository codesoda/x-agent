## Patterns & Conventions
- All runners use `#!/usr/bin/env bash`, `set -euo pipefail`, and parse `-h|--help|help` before any `need()` checks so help works without local project context.
- Shared defaults and helpers are inherited from `lib/x-agent-common.sh`; runner scripts remain mostly step-specific logic.
- `need <tool>` exits with code 2 on missing required dependency and prints `Missing required tool: <tool>`.
- Logging pattern is consistent: `setup_outdir` creates temp `OUTDIR`; per-step logs write to files there; `print_result` and `print_overall` emit final status and log path.
- Command dispatch follows `lint`, `format`, etc. with `all` as default; per-step toggles use `RUN_*` env knobs.
- CHANGED_FILES scoping is implemented directly in each runner’s discovery logic, not centrally.
- On failure or `KEEP_DIR=1`, runner exits preserving logs; otherwise temp log dir is removed.
- `FAIL_FAST=1` is represented in `setup` + `should_continue` checks in `all` execution path.
- SKIP behavior is explicit and returns success (`0`) with `Result: SKIP (...)` messages.
- Install and user docs update is part of every new agent implementation.

## Database Schema
- No application-level database.
- Work-state metadata lives in `.ralph/*.jsonl` and logs (agent audit artifacts only).

## Test Infrastructure
- `tests/run-scenarios.sh` runs scenario fixtures and returns failure on non-zero differences from `EXPECT_EXIT`.
- Harness pre-checks for required tools in `REQUIRED_TOOLS` and marks scenario as SKIP if unmet.
- Shellcheck runs on all `skills/*.sh` and `lib/*.sh` before scenarios; must pass with `--severity=warning`.
- Scenario fixtures are currently exit-code centric; richer assertions are done during planning/review when required by spec.
- No dedicated test framework beyond shell + fixture directories; coverage relies on fixture diversity and `RUN_ARGS` combinations.
- Existing scoped behavior fixtures commonly use exported `CHANGED_FILES` and separate scenario directories.

## Architecture & Data Flow
- Invocation flow: user/CI/agent framework -> script loads shared lib -> parses flags/help -> resolves target files -> runs command/steps -> writes per-step logs to OUTDIR -> emits `Step`/`Result`/`Fix`/`Overall` -> trap cleanup writes exit-surface.
- Dependency flow: agents rely on local developer tools (`sqlfluff`, etc.), no internal RPC or service calls.
- Locking flow: each runner acquires file lock on `<tmp>/<agent>-agent.lock` (flock or perl fallback), preventing concurrent contention.
- File discovery flow: full project traversal (`find`) when no `CHANGED_FILES`; constrained filtering when `CHANGED_FILES` is set.
- Security posture: local-only CLIs; primary risks are command execution on file paths and external tool output parsing.
