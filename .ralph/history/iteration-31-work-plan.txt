## Architecture Review

### System Design and Boundaries
- `docker-agent` is a thin orchestrator over local tools and discovery logic: it accepts CLI/env inputs, normalizes scope, runs one or more steps, and emits standardized output.
- Universal components come from `lib/x-agent-common.sh`.
- Agent-specific logic lives entirely in `skills/docker-agent/scripts/docker-agent.sh` and covers: knob defaults, discovery, two checks, output orchestration, and command routing.
- Metadata and integration points are external: `tests/run-scenarios.sh` (fixtures), `install.sh` (installation & discovery), and `README.md` / SKILL.md (operator contracts).

### Dependency Graph and Coupling
- Hard dependency: `hadolint` for `lint`.
- Optional dependency: `docker` for `build-check`.
- Runtime coupling: `CHANGED_FILES` and `RUN_*` are passed through environment and used in discovery/gating.
- Shared-lib coupling: any changes to `x-agent-common.sh` (especially `print_result`, `setup_outdir`, `setup_lock`, `need`) directly affect all agents.
- Installer coupling: missing `docker-agent` in `SKILLS` and dependency checks blocks user discoverability and installation behavior.

### Data Flow and Bottlenecks
- Data flow: Scenario/env → agent bootstrap → candidate discovery → per-step command execution → logs → structured step output → final exit code.
- Main bottlenecks are filesystem scans (`find`) and external command execution cost.
- `all` path is bounded by number of discovered Dockerfiles because each file runs one hadolint command (and one build-check command when enabled).

### Scaling and Single Points of Failure
- Largest scaling cost appears in broad repository scans; limit recursion to required extensions and exclude common non-project directories as in existing agents.
- Single points: missing `hadolint`, missing `docker` (for build-check), permissions/access on target repository, and lock file contention across concurrent runs.
- `setup_lock` in shared lib reduces run collisions but does not avoid tool contention (`docker`/`hadolint` runtime limits).

### Security Architecture
- No remote auth/API integration in docker-agent.
- External command execution surface is restricted to shelling out to configured binaries using repository-local file paths from discovery.
- Risk controls: consistent quoting, controlled candidate lists, and explicit scope filters reduce untrusted path expansion.

### Key Architectural Decisions

Decision 1: discovery strategy
- Option A (recommended): `find` recursion with explicit prune patterns and `case`/globs for Dockerfile names.
  - Effort: low, matches existing style. Risk: large trees still scanned.
- Option B: use `git ls-files` first and filter tracked files.
  - Effort: medium; misses untracked temporary fixture files and is inconsistent with other agents.
- Option C: delegate discovery to external index tool.
  - Effort: high and extra dependency; anti-pattern for this repo.
- Chosen: Option A.
- Implication: straightforward integration with existing patterns and no new dependency risk.

Decision 2: build-check scheduling
- Option A (default): include `build-check` only when `RUN_BUILD_CHECK=1` and not in clean default `all` path.
  - Effort: minimal and aligns with FRs.
- Option B: always run build-check in `all`.
  - Effort low but slow/expensive and can fail due environment.
- Option C: run build-check as part of `all` only when `docker` available.
  - Better defaults than B but still unexpectedly heavy.
- Chosen: Option A.
- Implication: prevents expensive defaults while keeping opt-in explicit.

Decision 3: logging granularity
- Option A (recommended): per-file command outputs appended to one step log, with summary `print_result` and truncated stdout.
  - Effort: medium; clear evidence and easy log discovery.
- Option B: one log file per file.
  - Effort high with cleanup/path noise.
- Option C: stream only final output, no full logs.
  - Effort low but weak diagnostics and rework-heavy.
- Chosen: Option A.
- Implication: keeps logs searchable and consistent with existing agents.

### Future Impact
- Decision choices preserve shared patterns used by upcoming 09/10 agents.
- `CHANGED_FILES` semantics and scoped fixture expectations should be reused as default for other discovery-based new agents.
- Explicit optional-step behavior (default-off `RUN_BUILD_CHECK`) sets a precedent for expensive operations in later agents.

## Code Quality Planning

### Organization and Module Structure
- Keep one script file under `skills/docker-agent/scripts/` with small helper functions.
- Maintain one-step-per-function structure: `discover_dockerfiles`, `run_lint`, `run_build_check`.
- Use `LIB_DIR` + shared-lib sourcing exactly once; avoid duplicating lock/cleanup/parsing helpers.

### DRY Opportunities and Reuse
- `step`, `fmt_elapsed`, `need`, `print_result`, `setup_outdir`, `setup_lock`, `print_overall` from common lib.
- Reuse `RUN_*` pattern and `should_continue` pattern from `all` command.
- Reuse discovery pattern style from `gha-agent` for `CHANGED_FILES` filtering.
- Avoid introducing new local utility duplicates for printing `Result/Time` that bypass common helpers.

### Error Handling and Edge Cases (explicit)
- Required tool missing (`hadolint`): fail fast with exit 2, clear install guidance.
- Optional `docker` missing while `RUN_BUILD_CHECK=1` or `build-check` command invoked: emit `Result: SKIP (docker not found)` for that step and continue with non-fatal semantics in scoped contexts.
- No Dockerfiles discovered: emit explicit `Result: SKIP` and treat as success.
- `CHANGED_FILES` set but no matches: skip with `Result: SKIP` and return success.
- `CHANGED_FILES` contains paths to non-Dockerfile entries: ignore safely.
- `docker build --check` exits non-zero for a file: fail step and continue aggregation.
- Log directory handling: rely on `setup_outdir` for cleanup semantics.

### Engineering Level and Maintainability
- Prefer predictable, explicit loops over compact one-liners to reduce shell ambiguity.
- Keep output messages deterministic and aligned with existing contract.
- Keep `docker-agent.sh` under ~200 LOC of domain-specific code by delegating boilerplate to the shared library.
- Add comments for decision-sensitive behavior (`RUN_BUILD_CHECK` default OFF, optional skip).

### Technical Debt Controls
- Document assumptions for expensive operations and file-globbing semantics in `SKILL.md` and script `usage()`.
- Keep scenario coverage close to FR requirements to prevent silent regressions (especially scope + skip branches).

## Test Strategy

### Coverage Scope
- Smoke/e2e: `tests/run-scenarios.sh docker-agent` (fixture-based).
- Contract tests for install/README consistency by manual inspection or explicit command checks in work plan run.
- Shell lint: `shellcheck --severity=warning` via scenario harness.

### Scenario Plan
- `tests/docker-agent/clean/` with valid Dockerfile and `REQUIRED_TOOLS="hadolint"` should pass.
- `tests/docker-agent/issues/` with hadolint violations should fail.
- Add `tests/docker-agent/scoped-match/` with `CHANGED_FILES` that matches one valid Dockerfile and `EXPECT_EXIT=0`.
- Add `tests/docker-agent/scoped-no-match/` with `CHANGED_FILES` pointing to unrelated files and `EXPECT_EXIT=0` to assert SKIP.
- Add `tests/docker-agent/no-dockerfiles/` for explicit no-target SKIP behavior if fixture directory is empty/contains non-matching files.
- Optional `tests/docker-agent/build-check-off/` to verify command defaults (no build-check when `RUN_BUILD_CHECK=0`).
- Optional `tests/docker-agent/build-check-skip/` with `RUN_BUILD_CHECK=1` and no `docker` binary via harness assumptions to assert SKIP message semantics (or document as environment-conditioned manual check if harness cannot stub tool removal).

### Failure Modes and Error Path Assertions
- Missing required tool: expect exit code 2 when `hadolint` absent.
- `lint` failure from file content violations: expect exit 1 and `Fix:` line with `/docker-agent lint`.
- `build-check` failure: expect exit 1 and `Fix:` line with `/docker-agent build-check`.
- `CHANGED_FILES` mismatch: expect SKIP and successful exit (0).

### Test Quality Rules
- Assertion strength should validate both behavior and message (exit code plus presence of explicit `Result:` reasons).
- Use existing harness semantics to avoid shell output brittle diffs.
- Keep Dockerfile fixtures minimal and deterministic.

## Performance Considerations

### N+1 and Command Cost
- No database access exists.
- Avoid repeated directory scans: discover once per command run and reuse candidate set across steps.
- Avoid nested `find` per file; do exactly one scan per invocation.

### Memory/State
- Use line-oriented strings/arrays for discovered file list; avoid holding full file contents.
- Keep logs stream-based and on disk; avoid in-memory accumulation of full tool output for all files.

### Caching and Reuse
- Cache discovery result in a variable within invocation.
- No cross-run cache to avoid stale results in mixed working trees.

### Slow Paths
- `find` on large monorepos and repeated `hadolint`/`docker build --check` runs are the two slow paths.
- Build-check is explicitly opt-in to protect default run-time.

### Optimization Priority
- Low: discovery prune patterns (`.git`, `node_modules`) only.
- Medium: avoid launching docker at all unless explicitly enabled.
- Optional: use one `hadolint` command per file vs one aggregate command; per-file execution allows exact failure attribution and easier mapping with explicit file-level results, and is acceptable for performance with expected fixture-scale workloads.

## Decision Log

- Decision: Keep discovery in Bash using `find` instead of git-index enumeration.
  - Options considered:
    - A) `find` + extension/path filters + CHANGED_FILES narrowing.
    - B) `git ls-files`-only discovery.
    - C) external indexing service.
  - Chosen approach: A.
  - Implications: deterministic across tracked/untracked files and no extra dependency; aligns with current agent patterns.

- Decision: Keep `build-check` optional and default off.
  - Options considered:
    - A) default off with `RUN_BUILD_CHECK=0`.
    - B) default on with expensive execution overhead.
    - C) remove command entirely from default flow.
  - Chosen approach: A.
  - Implications: faster default run; explicit opt-in controls side effects and required environment.

- Decision: Use `print_result` + centralized `hadolint` log per step.
  - Options considered:
    - A) shared helper with per-step full log.
    - B) per-file logs only.
    - C) no file logs.
  - Chosen approach: A.
  - Implications: consistent output contract and easier debugging in downstream workflows.

- Decision: Implement `CHANGED_FILES` semantics with explicit scoped match/no-match coverage.
  - Options considered:
    - A) only clean/issues fixtures.
    - B) add dedicated scoped fixtures for positive and negative matches.
    - C) no tests, trust code.
  - Chosen approach: B.
  - Implications: reduces regression risk for path-based discovery logic and aligns with existing architecture expectations.

### 6. Implement `build-check` step

How:

- For each Dockerfile, run `docker build --check -f "$dockerfile" .`.
- Keep logs in `${OUTDIR}/build-check.log`.
- Return aggregated exit status across all files.
Edge Cases:
- Docker daemon unavailable causes non-zero; do not hide and return failed step unless command requested skip condition.

## Edge Cases and Risks
- Discovery overreach in monorepos with many nested `.dockerignore`-like generated files.
- Paths with spaces in `CHANGED_FILES` can mis-parse if shell loops are plain-word split.
- `docker build --check` might require daemon/config absent in CI images.
- `hadolint` output format changes can affect regex-free log expectations.
- Optional-tool behavior could be misread as global failure when only `build-check` is failing but `lint` passes.
- `SKIP` vs `FAIL` semantics must remain explicit in each branch to satisfy exit-code expectations.

## Recommendations for Immediate Implementation Quality
- Preserve command output contract exactly; use shared helpers for every result boundary.
- Keep skip semantics explicit and deterministic (`SKIP` on no inputs, no-docker, and empty scope).
- Use scoped fixtures in addition to base clean/issues so `CHANGED_FILES` behavior is not under-tested.
- Keep `all` cheap by default with `RUN_BUILD_CHECK=0`.

## Lessons Applied (for this plan)
- Reuse existing fixture discipline for environment export requirements (`CHANGED_FILES`, optional knobs) before execution.
- Prefer explicit `SKIP` messaging branches for discoverability and easier fixture assertions.
