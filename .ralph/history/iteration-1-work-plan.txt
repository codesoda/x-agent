## Architecture Review

## 1. Overall system design and component boundaries
The current architecture is a thin CLI shell runner model: each agent script is a thin orchestrator and each step is a local workflow check. The shared concerns (output formatting, temp directory handling, fail-fast behavior, dependency checks, locking, and lock-aware lifecycle) are duplicated in each script. Spec 01 introduces a new library boundary: `lib/x-agent-common.sh` becomes the reusable runtime substrate used by all agents while each agent keeps domain-specific command execution only.

Component boundaries become:
- Agent script: project detection, command translation, and tool invocation per stack.
- Shared common library: process lifecycle primitives, diagnostics helpers, lock, cleanup, output contract primitives.
- Installer: packages scripts and shared library into a source-consumable layout.
- Scenario harness: verifies behavior, exit codes, and fixture outputs without mocking internal helpers.

This separation reduces drift because all agents should call the same primitives and maintain consistent contract.

## 2. Dependency graph and coupling concerns
Primary dependencies are: `agent script -> lib/x-agent-common.sh`, `install.sh -> lib/x-agent-common.sh` packaging, and `tests/run-scenarios.sh -> agent scripts + fixture files`. Inconsistent path assumptions are the highest coupling risk, because the common library is external to each agent file and loaded by relative path.

Current coupling points:
- Lock behavior relies on shared temp directory location (`TMPDIR_ROOT`) and file descriptor availability.
- Output contract relies on shared `Result/Overall` printers.
- Installer path topology relies on script-directory-relative resolution for `lib/`.
- Scenario runner expects each `AGENT_SCRIPT` to continue emitting expected `overall` format and exit semantics.

Decoupling strategy:
- Keep common library API deliberately small and explicit.
- Keep lock helper name collisions impossible (single shared helper names used consistently).
- Push all path resolution logic into a predictable function.

## 3. Data flow patterns and potential bottlenecks
Normal data flow:
1. Agent entry → parse flags/env → set defaults.
2. Source library → initialize output and optional lock state.
3. Run per-command steps, each writing logs to `OUTDIR`.
4. Emit bounded diagnostics → print step result → aggregate `overall_ok`.
5. Print overall summary and return aggregate status.

Bottlenecks are low and mostly I/O: startup overhead for logs and lock coordination. The heaviest path is concurrent lock contention and large tool output captured by shell redirection. The common library should avoid extra command invocations and avoid scanning whole projects unless already part of domain steps.

## 4. Scaling characteristics and single points of failure
This codebase has low horizontal scale but high invocation frequency, especially in CI and agent loops. The major scaling limitation is lock serialization by design: one agent run per tool at a time to prevent nondeterministic workspace corruption. This is acceptable and expected.

Single points of failure:
- Incorrect lock implementation can deadlock/hang all concurrent runs.
- Bad `LIB_DIR` resolution breaks all agents in one shot.
- `cleanup` semantics that delete logs too aggressively remove debug data on failure.
- Installer remote path copy omission can create runtime `No such file` for all installed agents.

## 5. Security architecture
The project runs local command-line tooling; there is no remote API auth in this spec. Security concerns are process-local:
- Avoid path traversal and unsafe eval in source loading.
- Keep `source`ing limited to trusted repo-controlled path and preserve shellcheck source hints.
- Treat `AGENT_SCRIPT` and tool outputs as untrusted fixture input only in tests; production runs are local.
- Lock file and temp dirs should stay under writable temp roots to reduce cross-project interference.

No credentials are embedded; security hardening is mostly filesystem boundary and deterministic path assumptions.

## Key Architectural Decisions

## Option set A: Minimal helper extraction
- Keep library as one file containing only currently duplicated functions/vars.
- Pros: low effort, low risk.
- Cons: weak future reuse for lock/install path variants.
- Risk: repeated remote install workarounds in scripts.

## Option set B: Full bootstrap utility module in library (recommended)
- Add lifecycle + lock + path resolution helpers plus optional test helper function (small private helpers).
- Pros: one canonical source for setup correctness; lowers spec 02-10 refactor cost; future agents only source + call.
- Cons: larger surface, needs clearer docs and stricter tests.
- Impact on future specs: simplifies future agent onboarding.

## Option set C: Keep lock/resolve helpers in install script only
- Move install-specific path logic fully into `install.sh` and keep library generic only.
- Pros: less library responsibility.
- Cons: agents still depend on script-specific assumptions; brittle for local vs remote and symlink contexts.
- Impact on future specs: likely repeated edge-case fixes in each new agent.

Recommended: Option B, because it aligns with the spec statement and keeps repeated orchestration concerns in one place.

## Code Quality Planning

## Module organization
Use a single `lib/x-agent-common.sh` file with the exported functions below and no domain logic:
- variable initialization helpers
- `need`, `hr`, `step`, `fmt_elapsed`, `should_continue`
- `setup_outdir`, `setup_lock`, `print_overall`, `print_result`
- optional `resolve_lib_dir` helper

No other files are modified in spec 01 for existing agent logic.

## DRY opportunities
The top-level duplicated blocks are exactly the nine items in FR implementation details. This spec should stop re-duplicating:
- cleanup trap setup
- max-lines initialization
- step timing logic
- failure handling
- required tool guard
- flock/perl lock fallback

Potential future drift points where reuse is not yet present: CLI parsing of `--help` and `--fail-fast` and common usage text structure.

## Error handling plan
- Missing required dependency: `need()` must emit to stderr and exit `2` immediately.
- `setup_outdir` failures (`mktemp`): rely on `set -e` for hard failure; ensure `trap` can read `OUTDIR` safely and not reference empty variable.
- Lock setup failures:
  - `flock` missing + `perl` missing: emit warning and continue unlocked.
  - lock acquisition contention: print deterministic waiting message before blocking.
- `setup_lock` called before lock-sensitive work but after trap installation to cover setup failure paths.
- `print_result` must always print fix lines only on FAIL and include `Full log` path.

## Technical debt prevention
- Keep shell features Bash 3.2 compatible (no arrays requiring newer semantics).
- Explicitly centralize all default env knob handling in lib.
- Add shellcheck directives and local variable naming conventions to avoid unbound errors.
- Keep lib side-effect-free by design.

## Engineering level
Avoid over-architecting: this is one small compatibility-focused shared module with minimal public surface. Do not introduce abstractions like generic plugin maps or complex option parsing, which would increase fragility.

## Explicit code style choices
- Use plain functions and local variables.
- Keep command quoting explicit.
- Use predictable function names and fixed `overall_ok` contract from caller.

## Test Strategy

## Coverage layers
- Unit-style shell checks: load `lib/x-agent-common.sh` and verify exported functions are callable.
- Integration checks: run representative command paths against a tiny scratch script using `setup_outdir`, `print_result`, lock helper, and `print_overall`.
- E2E compatibility checks: ensure existing `tests/run-scenarios.sh` suites still pass for npm/cargo/terra with unchanged scripts.

## Critical edge cases to cover
- CI max lines behavior (`CI=true`, `CI=1`, and local default).
- `need` with missing command returns exit code 2.
- `setup_outdir` with `KEEP_DIR=1` and with failing script behavior.
- lock fallback behavior when `flock` missing, when `perl` present, and when both are missing.
- sourcing library file must produce zero output and no trap registration.
- printing function on PASS and FAIL must include required fields and fix hints.
- remote-path scenario: `resolve_lib_dir` resolving to installed root when lib is copied.

## Failure modes and explicit assertions
- If lock cannot be acquired while another run holds it, output must include “waiting for another run to finish” message and complete after lock release.
- If required tool missing, agent invocation should exit with 2 before work begins.
- If helper trap fails due to missing `OUTDIR`, script should fail fast rather than silently hide cleanup errors.
- If `print_result` is asked to process FAIL without hint, tests should fail because spec requires fix guidance.

## Quality of assertions
Assertions must check both status code and textual fields (`Overall: PASS|FAIL`, `Logs:` lines, `Full log`, `Fix:`). For lock checks, assert message ordering where practical.

## Performance Considerations

## I/O and process behavior
- Most commands are external tool invocations (`cargo`, `terraform`, `node`), already outside this spec.
- The shared helper layer should stay lightweight to avoid adding startup cost to every agent run.

## N+1 / repeated scans
No database queries exist in this repository. Avoid introducing additional repeated filesystem scans in helper functions beyond what is already required by the caller domain logic.

## Memory usage
Use small scalars/arrays only; avoid building large logs in shell arrays.

## Caching opportunities
No shared cache needed. If future needs arise, cache resolved `LIB_DIR` in a local variable once.

## High-latency paths
- Lock waits are expected to block; add clear message and keep waiting bounded by external process lifecycle.
- `mktemp` and temporary log writing are cheap relative to domain commands.

## Hotspots requiring optimization
Only lock contention can serialize throughput; keep lock scope minimal (acquire once at startup, hold only process-lifetime).

## Implementation Plan

## Step 1 — Establish a sourceable common library contract
- What: Create `lib/x-agent-common.sh` with `#!/usr/bin/env bash` and no auto-executed functions.
- Why: The spec requires additive extraction and no side effects on source.
- How: Define only reusable primitives and defaults; include comments that this file is sourced.
- Edge Cases: If sourced from legacy shells lacking Bash 3.2 features, avoid unsupported constructs.
- Tests: `source` the file under a clean shell and assert no stderr/stdout output.
- Risks: introducing hidden side effects through traps or command substitutions.

## Step 2 — Centralize shared environment defaults and helper functions
- What: Add `KEEP_DIR`, `FAIL_FAST`, `CHANGED_FILES`, `TMPDIR_ROOT`, and CI-aware `MAX_LINES` defaults in library.
- Why: Removes duplicated env handling and ensures behavioral consistency.
- How: Use the exact CI defaults from spec; allow explicit `MAX_LINES` override precedence.
- Edge Cases: `CI` can be unset, `CI=true`, `CI=1`, or other values.
- Tests: Simulate local and CI envs and compare output defaults.
- Risks: wrong precedence can make CLI overrides ineffective.

## Step 3 — Implement lifecycle helpers
- What: Add `hr`, `step`, `fmt_elapsed`, `should_continue`, `need` and `resolve_lib_dir`.
- Why: these are the hard extraction points shared across all three agents.
- How: Keep function names and signatures stable with caller expectations.
- Edge Cases: `need` must exit `2`; `should_continue` must honor `FAIL_FAST` + `overall_ok` state.
- Tests: assert each function output and exit status with synthetic callers.
- Risks: breaking existing `overall_ok` guard if `should_continue` semantics drift.

## Step 4 — Implement setup helpers with trap safety and lock support
- What: Implement `setup_outdir` and `setup_lock`.
- Why: setup functions are required and currently duplicated across scripts.
- How:
  - `setup_outdir`: allocate `mktemp` dir, set `OUTDIR`, install trap function that preserves logs on failure or `KEEP_DIR=1`.
  - `setup_lock`: `flock -n` path with perl fallback and deterministic waiting message.
- Edge Cases:
  - `TMPDIR_ROOT` unwritable.
  - lock helpers unavailable.
  - multiple shell invocations in same process.
- Tests:
  - run `KEEP_DIR=0` success and verify dir deleted by default.
  - run simulated fail and verify dir preserved.
  - lock contention simulation using helper script in same pattern as existing cargo fixture.
- Risks: lock fallback syntax differences across macOS/Linux; ensure POSIX-compatible fallback path is stable.

## Step 5 — Implement result/overall formatting helpers
- What: Add `print_result` and `print_overall`.
- Why: centralize Step/Result/Fix/Overall contract while preserving behavior.
- How: Ensure `print_result` prints `Full log` and `Time`; print fix hint only when `ok` is false.
- Edge Cases: `ok` value malformed or empty; guard by explicit parameter checks.
- Tests: compare function output format against expected substrings.
- Risks: agents might pass pre-formatted multiline fix hints; keep caller flexibility.

## Step 6 — Add install-time path strategy and remote accessibility
- What: Update `install.sh` to include shared library packaging for remote installs and ensure local installs remain unchanged.
- Why: `lib/x-agent-common.sh` must be accessible from installed skill scripts.
- How:
  - For remote mode, fetch or copy `lib/x-agent-common.sh` into `SOURCE_DIR/lib` and/or per-destination `root/lib` before runtime.
  - Keep local mode symlink behavior so no extra copy is required.
  - Keep `patch_skill_paths` scoped to scripts paths only, while adding docs/notes for required shared lib location.
- Edge Cases: local `install.sh` run from repo root vs script directory; mixed CODEX/Claude installs; SKILLS filtered by user selection.
- Tests: perform a dry-run style check by sourcing an installed script path with simulated `SCRIPT_DIR` relative resolution.
- Risks: remote mode copies stale lib if fetch fails; add explicit fetch error handling.

## Step 7 — Validate compatibility with unchanged agents
- What: Run `tests/run-scenarios.sh` after only shared library and installer changes.
- Why: acceptance for spec 01 is no behavior regressions in existing agents.
- How: execute full scenario run and targeted run by agent; do not modify existing scripts.
- Edge Cases: lock timing race in cargo fixture; transient CI dependency differences.
- Tests: `tests/run-scenarios.sh cargo-agent`, `tests/run-scenarios.sh npm-agent`, `tests/run-scenarios.sh terra-agent`, plus full run.
- Risks: hidden fixture failure due to lock helper behavior or cleanup changes.

## Decision Log

## Decision 01 — Keep `lib/x-agent-common.sh` as additive source-only module
- Options considered:
  - `1A`: Add only extracted boilerplate and leave installer untouched.
  - `1B`: Add full setup + path resolution + install compatibility.
  - `1C`: Skip shared lib and keep ad-hoc scripts.
- Chosen approach: `1B`.
- Implication: lowers churn in future specs by standardizing both runtime and packaging assumptions.

## Decision 02 — Use file-based lock with flock + Perl fallback
- Options considered:
  - `2A`: `mkdir`-based lock directory.
  - `2B`: `flock` plus Perl fallback as in existing agents.
  - `2C`: No lock in shared library.
- Chosen approach: `2B`.
- Implication: behavior remains aligned with existing agent semantics and lock test patterns.

## Decision 03 — Install-time strategy for shared lib
- Options considered:
  - `3A`: Always symlink entire repo into remote installs.
  - `3B`: Copy only lib into install root and keep skill directories separate.
  - `3C`: Bake absolute path inside each script.
- Chosen approach: `3B`.
- Implication: minimal install payload and deterministic resolution from `skills/<name>/scripts/../../../lib`.

## Edge Cases and Risks

- `KEEP_DIR` precedence: must preserve logs on success only when explicitly requested.
- Double-trap conflicts if caller already defines `trap` handlers; document one-time setup expectation.
- `mktemp` template collisions when `TMPDIR_ROOT` is relative or points to read-only FS.
- Lock collision where script exits before releasing fd 9 (ensured by shell exit trap semantics, but must not override trap unintentionally).
- `SCRIPT_DIR` path resolution in spaces-containing paths.
- `resolve_lib_dir` returning non-existent path in remote install due to partial copy.
- Installer user selection skipping all skills should still complete cleanly.
- CI environment differences where `CI` and shell settings differ by runner.
- Future spec 02 refactors may call `setup_lock` from scripts that do not need lock; safe no-op behavior should remain.

## Risks and dependencies

- Dependency on shell behavior across Bash 3.2 and later.
- Installer relies on shell `cp -R` behavior for permissions and symbolic links.
- `tests/run-scenarios.sh` currently checks scripts under `skills/*` and may not include `lib` directory; this is acceptable but maintain script sourceability.
- No new external dependencies introduced.
