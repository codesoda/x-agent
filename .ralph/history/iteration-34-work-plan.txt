## Architecture Review

### System Design and Component Boundaries
- The ansible-agent remains a thin orchestration shell around two command steps (`lint`, `syntax`) and is invoked directly by `run-scenarios.sh` fixtures.
- Shared behavior (`step`, `print_result`, `setup_outdir`, `setup_lock`, `need`, etc.) comes from `lib/x-agent-common.sh`; ansible-agent contributes only domain-specific discovery and command execution.
- The script boundary is: `run-scenarios.sh` -> ansible-agent -> local CLI tools (`ansible-lint`, `ansible-playbook`) -> filesystem reads + OUTDIR logs.
- External integration points are minimal: `install.sh` for discoverability and `README.md` for user contract; no runtime service calls.

### Dependency Graph and Coupling
- Required runtime deps: `ansible-lint` and `ansible-playbook` (`need`-guarded and checked before running steps).
- Shared lib coupling is critical: changes to `x-agent-common.sh` (especially `need`, `print_result`, lock logic) impact all agents and should remain backward-compatible.
- Test coupling is through `.env` contract keys consumed by the harness (`SCENARIO_NAME`, `AGENT_SCRIPT`, `EXPECT_EXIT`, `REQUIRED_TOOLS`, `RUN_ARGS`, optional `CHANGED_FILES`).
- UI coupling via `SKILL.md` and README influences agent discoverability and automation matching quality.

### Data Flow and Bottlenecks
- Flow: parse CLI/help/flags -> set defaults -> source/discover tool availability -> initialize output dir + lock -> resolve mode/targets -> execute command(s) -> write per-step logs -> print Step/Result/Fix/Overall.
- CHANGED_FILES affects two branches differently: scope decision for syntax-check targets and skip decision for lint run availability.
- Primary runtime bottlenecks: recursive YAML discovery (`find`) and per-playbook `ansible-playbook --syntax-check` execution in large repos.

### Scaling Characteristics and Single Points of Failure
- In monorepos, recursive discovery dominates; prune rules are needed to avoid scanning binaries/node_modules equivalents.
- Single points of failure: missing required tools (`need` -> exit 2), no playbook discovery (`SKIP` path), malformed YAML causing syntax-check exits, or corrupted lock/temp directory handling.
- Locking in shared lib avoids concurrent tempdir/log interference but does not prevent command-level contention of ansible tooling.

### Security Architecture
- No remote auth/API boundaries are introduced.
- Main attack surface is filesystem path handling and shell command execution; use guarded file filtering and quoting to prevent globbing issues.
- Running `ansible-lint --fix` locally is mutating by design; CI must force check mode and keep mutation-free behavior.

### Key Architectural Decisions

Decision 1: Playbook discovery model
- Option A (recommended): recursive `.yml/.yaml` candidate scan + filename heuristics + standard directories + optional `hosts:` filter for syntax candidates.
- Option B: only `ansible-lint --list`/CLI-discovered paths (simpler, lower control over scoped syntax path mapping).
- Option C: parse YAML safely with external parsers (`yq`, `python`) to infer playbook structure (accurate but adds dependencies and complexity).
- Chosen: Option A.
- Why: deterministic, fast enough for repo-scale fixture patterns, and consistent with existing agents' discovery style.
- Future impact: this heuristic can be reused for sql/kube-like agents needing mixed path+content rules.

Decision 2: Scope semantics for lint vs syntax under `CHANGED_FILES`
- Option A (spec-compliant, recommended): syntax is scoped to changed playbook files; lint runs project-wide, but with scoped skip if no relevant YAML changed.
- Option B: scope lint to changed files only.
- Option C: scope lint to changed files only for changed-only mode and all files otherwise.
- Chosen: Option A.
- Why: matches FR-3/CHANGED behavior and avoids false negatives from partial lint decisions.
- Future impact: keeps parity with multi-step agents where expensive/cheap steps can have different scope semantics.

Decision 3: FMT_MODE handling
- Option A (recommended): normalize `FMT_MODE` once in `resolve_fmt_mode` (`auto` -> `fix` local, `check` in CI).
- Option B: call `ansible-lint --fix` always and skip CI override.
- Option C: only `check|fix` allowed, no CI override.
- Chosen: Option A.
- Why: aligns with go-agent/terra-agent conventions and FR-10.
- Future impact: low-friction upgrade path for future agents needing local auto-fix semantics.

## Code Quality Planning

### Organization and Module Structure
- Keep file small and domain-specific (~<200 lines): discovery helpers, mode resolution, two step runners, and one CLI main entrypoint.
- Group variables in top block: universal knobs, agent knobs, and usage metadata.
- Keep `usage()` and `collect_playbooks` simple and testable by fixture output.

### DRY Opportunities and Repetition Risk
- Reuse `print_result` and common timing/cleanup behavior from shared lib instead of custom equivalents.
- Centralize shared skip branch text in small helper constants if needed (`skip_no_targets`, `skip_no_syntax_targets`) to avoid copy drift.
- Use one discovery path variable and share filtering operations to prevent duplicated extension checks.

### Error Handling Patterns (Explicit)
- `need ansible-lint` and `need ansible-playbook` happen before execution to fail fast with code 2.
- `resolve_fmt_mode` validates unsupported values and exits 2 with actionable message.
- Discovery functions emit explicit zero-target reasons and return success (no crash), letting caller generate deterministic `SKIP`.
- Command runners capture command exit codes, dump full output to logs, and only print truncated tails on failures.
- `main` maps unknown commands to usage + exit 2.

### Technical Debt Controls
- Avoid clever parsing of YAML; use conservative heuristics with explicit limitations documented in usage and SKILL.
- Keep command args in arrays where practical; avoid broad global state and deeply nested logic.
- Add explicit fixture coverage for every branch with `CHANGED_FILES`, `all`, and command-specific invocation.
- Prefer plain POSIX/Bash 3.2 constructs (no associative arrays/readarray/
  shell features outside compatibility bounds).

### Engineering Level
- Engineered enough: explicit branching for scope and modes, not over-abstracted.
- Avoid over-engineering by not adding config files, caching services, or language parsers unless future iterations need richer semantics.

## Test Strategy

### Coverage Plan
- Unit-like validation (script-level behavior)
  - `bash -n` syntax on script and shellcheck severity warning checks in harness.
- Integration (scenario-based)
  - `tests/run-scenarios.sh ansible-agent` as the canonical acceptance harness.
- End-to-end contract checks
  - Validate `install.sh` registration and README/Skill discoverability by running scenario coverage and grepping for required references.

### Required Edge Cases + Fixture/Assertion Mapping
1. **Clean pass path**: `tests/ansible-agent/clean/` with valid playbook and required tools. Expect exit 0.
2. **Issues fail path**: `tests/ansible-agent/issues/` with lint violations. Expect exit 1 and `Fix:` present.
3. **All command default path**: `RUN_ARGS="all"` on clean and issues fixtures.
4. **Lint-only command**: `RUN_ARGS="lint"` on clean/issues; confirms command dispatch and `lint` message/fix hint.
5. **Syntax-only command**: `RUN_ARGS="syntax"` with valid playbook + invalid syntax fixture to verify syntax discovery and aggregate behavior.
6. **CHANGED_FILES match (lint)**: export one valid playbook in `CHANGED_FILES`; expect lint step execute and overall pass/fail according fixture.
7. **CHANGED_FILES match (syntax)**: same but with `RUN_ARGS="syntax"`; expect only changed playbooks syntax-checked.
8. **CHANGED_FILES no-match**: export non-YAML or non-playbook paths; expect `Result: SKIP (no playbooks ... CHANGED_FILES)` and exit 0.
9. **No playbooks discovered**: fixture with only non-playbook YAML or no YAML files; all commands should `SKIP` gracefully with exit 0.
10. **FMT_MODE local fix branch**: `FMT_MODE=fix` in clean/issues fixture with local env; ensure command switches to `ansible-lint --fix`.
11. **CI forces check**: `CI=true FMT_MODE=fix` still executes check mode; fail path should still emit lint SKIP/Fix guidance for check mode.
12. **Tool failure path (manual)**: invoke script where `ansible-lint` missing to confirm exit 2 (via direct call or environment without tool, since harness marks missing as `SKIP`).

### Failure Modes and Error Path Testing
- Missing required binary -> exit 2 and clear `Missing required tool` message.
- Discovery misses files with scoped input -> SKIP path, code 0.
- ansible-lint returns error -> code 1 and fix hints with exact `/ansible-agent lint` command.
- ansible-playbook syntax failure -> code 1 with per-path logs and fix hint `/ansible-agent syntax`.
- Invalid `FMT_MODE` -> code 2 with explicit input guidance.

### Assertion Strength
- Validate both exit status and branch intent in scenario labels/messages:
  - `Result: SKIP` vs `Result: PASS/FAIL`
  - Fix hint content includes exact command path (`/ansible-agent lint`, `/ansible-agent syntax`).
- For non-empty failure cases, assert full log generation path appears in output.

## Performance Considerations

### N+1 and Command Cost
- Avoid recomputing discovery for both steps; compute once per run and reuse in `all`.
- Do not call `find` per playbook. One repository scan + per-target syntax calls is the upper bound.
- Prefer one `ansible-lint` invocation for lint step vs repeated per-file invocation.

### Memory Usage
- Store candidate paths in newline-separated strings or Bash arrays only as needed; avoid reading full file contents.
- Keep full diagnostics in disk logs under `OUTDIR`; avoid loading huge logs into memory.

### Caching Opportunities
- Cache discovery output for current invocation in environment variable(s) (`PLAYBOOKS`, `CHANGED_PLAYBOOKS`).
- No cross-run cache; stale discovery state across changing worktrees is riskier than repeated scans.

### Slow/Complex Paths
- Syntax-check is O(n files) process spawn overhead; this is expected and can be heavy for monorepos.
- `ansible-lint --fix` can be expensive and mutating; keep this local-only and CI check-only.

### Performance Monitoring
- If ansible repos become large, prioritize adding prune paths (e.g., `.venv`, `.git`, `.cache`) and optionally role/playbook root detection to limit initial scan depth.

## Decision Log

- Decision: Use `ansible-lint` for lint and `ansible-playbook --syntax-check` for syntax with shared skip semantics
  - Options considered:
    - A) Dedicated ansible parser + custom checks.
    - B) `ansible-lint` + `ansible-playbook --syntax-check` with heuristic discovery.
    - C) Single command strategy using only `ansible-lint`.
  - Chosen approach: B.
  - Implications: clear separation of semantic concerns and predictable error messages across commands.

- Decision: Keep syntax discovery stricter than lint discovery
  - Options considered:
    - A) Same target set for lint and syntax.
    - B) Syntax only runs files with explicit `hosts:` after YAML discovery.
    - C) Syntax uses lint-only target set.
  - Chosen approach: B.
  - Implications: reduces false syntax-check attempts on non-playbook YAML files and lowers avoidable command failures.

- Decision: Keep install/docs as required spec output artifacts, not optional
  - Options considered:
    - A) Add script+fixtures only.
    - B) Add script/fixtures + install/README.
    - C) Add docs and install updates in separate follow-up.
  - Chosen approach: B.
  - Implications: operator and automation discoverability remains accurate immediately after implementation.

## Edge Cases and Risks
- `CHANGED_FILES` contains spaces: shell word-splitting is path-sensitive; use tested patterns and document limitation.
- `CHANGED_FILES` includes only role helpers/templates without play-level hosts: syntax may skip while lint still executes.
- `ansible-lint --fix` may rewrite files; plan for fixture cleanup in manual run scripts if local mutation is tested.
- Syntax checking on large playbook trees can become slow; keep default `all` command path optimized with cached discovery.
- `ansible-playbook` on some enterprise collections may attempt optional dependencies; fixture design should avoid unavailable collection imports.
- `Need` checks can block execution in CI if tools are absent; fixture `REQUIRED_TOOLS` should stay explicit.
- Script output should preserve Step/Result/Fix/Overall contract even when SKIP paths occur.

**Edge Cases**: `source` path errors if script moved; rely on `$(cd ...)` path normalization.

**Risks**: None.

### 3. Implement `resolve_fmt_mode` and discovery helpers
**What**: Implement CI-aware mode resolver and helper functions for candidate detection.
**Why**: Centralizes behavior and avoids spread conditionals in runners.
**How**:
- `resolve_fmt_mode`: `auto -> local fix, CI check`; invalid mode -> exit 2.
- `collect_ansible_files`: gather existing `.yml/.yaml` files from `CHANGED_FILES` when set.
- `discover_playbooks`: recurse for filenames and role/playbook directories per spec; parse only files with `hosts:` for syntax list.
- Log how many matches were found.
**Edge Cases**:
- CHANGED_FILES includes non-existing files.
- No files found in repo scan.
- Commented `hosts:` lines not used for syntax selection.
**Tests**: scoped/ no-match fixtures.
**Risks**: Overly strict `hosts:` filter skipping valid playbooks with nonstandard layout.

### 4. Implement `run_lint` with project-wide checks and local/CI mode behavior
**What**: Add lint step using `ansible-lint`.
**Why**: Core requested linting functionality with fixed/mutating modes.
**How**:
- If CHANGED_FILES exists and no `.yml/.yaml` are relevant, emit SKIP.
- Else call `ansible-lint` with no positional arguments for full project discovery, using `--fix` only in local fix mode.
- Log output and emit `/ansible-agent lint` fix hint variations for check vs fix failures.
**Edge Cases**:
- `ansible-lint --fix` may modify files; run in local branches only.
- Lint may fail due syntax/parser errors in changed fixtures.
**Tests**: clean/issues, lint-only mode, FMT_MODE fix and CI check forced path.
**Risks**: Tool output volume; ensure `MAX_LINES` truncation and full log path.

### 5. Implement `run_syntax` over syntax-discovered targets
**What**: Add syntax step with `ansible-playbook --syntax-check` per playbook.
**Why**: Provides execution validation not covered by lint.
**How**:
- Iterate over `PLAYBOOKS_FOR_SYNTAX` list, run syntax-check per file into one log.
- On failures, track aggregate exit and print failing output slices.
- Return `SKIP` when no matching syntax targets.
**Edge Cases**:
- CHANGED_FILES empty + no syntax candidates.
- One target failing should fail whole step but still run no more files after? (decide fail-fast behavior is per-step only).
**Tests**: syntax-only pass/fail fixture.
**Risks**: Additional process overhead in large trees; acceptable with shared cache and path filters.

### 6. Implement command dispatch and all-mode orchestration

**Why**: Ensures compatibility with existing usage contract.

- `all` runs `run_lint` then `run_syntax` based on `RUN_*` with `should_continue`.
