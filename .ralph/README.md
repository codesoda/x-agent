# RALPH Loop - Project Guide

**Welcome to your RALPH (Recursive Autonomous Loop with Planning & Human-oversight) development environment!**

This project uses an autonomous AI development loop that breaks down requirements into discrete specifications and implements them iteratively with automated planning, implementation, and review cycles.

## Key Innovation

Each iteration starts with **fresh context** - no accumulated conversation noise. State persists through files, not chat history.

## Running the Loop

### Basic Usage

```bash
# Run the full autonomous loop
ralph loop

# Run a single phase (PLAN, WORK, or REVIEW)
ralph run
```

### Monitor Progress

```bash
# Check current status
ralph status

# View iteration history
ralph summary

# Detailed breakdown
ralph explain
```

### Configuration

Configure agent selection and timeouts in `.ralph/config.toml`:

```toml
plan_agent = "codex"    # or "claude"
work_agent = "claude"   # or "codex"
review_agent = "codex"  # or "claude"

max_iterations = 10
phase_timeout_secs = 1800
hang_timeout_secs = 300
```

Or use environment variables (take precedence over config files).

## Why This Approach?

### Codex for Planning & Review
- Strategic thinking and architectural decisions
- Critical evaluation and quality assessment
- Catches issues before they compound

### Claude Code for Implementation
- Detailed code generation
- Test writing
- Quick iteration on fixes

### Fresh Context Each Iteration
- No conversation bloat
- Clean slate for each task
- State persists through files

### Tight Feedback Loop
- Test early, test often
- Issues caught immediately
- No big-bang integration

## Troubleshooting

### Loop Stuck on REVISE?

Check review result and feedback:
```bash
cat .ralph/review-result.txt
```

Run suggested verification commands manually.

### Need to Debug?

View full logs:
```bash
cat .ralph/plan-agent.log
cat .ralph/work-agent.log
cat .ralph/review-agent.log
```

## Best Practices

1. **Keep specs focused** - Each spec should be completable in 1-3 iterations
2. **Write good acceptance criteria** - Clear, testable, objective
3. **Trust the process** - Let REVISE cycles happen, they improve quality
4. **Review feedback carefully** - Each REVISE makes the next iteration better
5. **Don't skip tests** - They're your safety net
