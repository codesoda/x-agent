## Architecture Review

### System Design and Component Boundaries
`helm-agent` should be a domain-specific command wrapper that sits in the same agent runtime model as existing runners:
- `skills/helm-agent/scripts/helm-agent.sh` owns command parsing, chart discovery, per-chart command execution, and output rendering.
- `lib/x-agent-common.sh` owns reusable mechanics: output framing, timing, trap/cleanup, lock, fail-fast guard, and requirement checks.
- `.ralph/specs/06-helm-agent.md` defines FR-level behavior and required scenarios.
- `install.sh` and `README.md` are registration/presentation boundaries.
- `tests/helm-agent` drives deterministic validation via `tests/run-scenarios.sh`.

The data path is file-system local and command-driven; no network/API calls are required in script behavior.

### Dependency Graph and Coupling Concerns
- Direct runtime dependency: `helm` binary in PATH.
- Filesystem dependency: chart discovery relies on `Chart.yaml` existence and relative parent directories.
- Tooling dependency in execution environment: `tests/run-scenarios.sh` and `shellcheck`.
- Cross-module coupling points:
  - install path assumptions in `install.sh` (`skills/<name>/SKILL.md`, `skills/<name>/scripts/<name>-agent.sh`).
  - shared lib location contract `../../../lib`.
  - SKILL allowed-tools contract used by skill runner policies.

Risk focus: brittle path assumptions (relative paths, discovery roots, and CHANGED_FILES expansion). Keep to existing convention of space-delimited CHANGED_FILES entries and explicit filtering logic to avoid regressions.

### Data Flow and Bottlenecks
1. Script invoked from fixture/project directory.
2. Parse help flags and command.
3. Validate helm tool.
4. Initialize `OUTDIR` and lock via common lib.
5. Resolve chart target list.
6. Execute one or more steps against each target, writing step logs.
7. Aggregate step outcomes into `overall_ok`, emit `Overall`.

Bottlenecks:
- `find . -name Chart.yaml` in large repos can be O(n).
- `helm lint`/`helm template` runtime scales with chart count and template size.
- CHANGED_FILES-driven execution can reduce runtime when scope is narrow.

### Scaling Characteristics and Single Points of Failure
- Single-tenant lock (`setup_lock`) avoids concurrent overlapping runs per agent name.
- Without CHANGED_FILES scoping, full-tree discovery and linting can be expensive on large monorepos.
- Critical failure surfaces:
  - invalid `CHART_DIR` path,
  - no chart roots found,
  - malformed chart files causing command exit >0.

### Security Architecture
- Threat surface is local filesystem path inputs (`CHART_DIR`, `CHANGED_FILES`).
- Security posture should remain non-networked and deterministic:
  - run only trusted local `helm` binary
  - quote paths when invoking commands (arrays when possible)
  - limit globbing/splitting assumptions to conventional conventions (already repo-wide).

### Key Architectural Options

Decision A: Chart root discovery strategy
- Option 1: Always scan entire repository for `Chart.yaml` and then apply CHANGED_FILES filtering.
  - Effort: low.
  - Risk: unnecessary scan and noisy behavior in huge trees.
  - Impact: easy to implement but less efficient.
- Option 2: If CHANGED_FILES is set, resolve only parent chart dirs first; else scan.
  - Effort: low.
  - Risk: missed chart when changed files are from untracked/deleted paths (acceptable when using best effort).
  - Impact: faster by default in incremental mode, matches FR intent.
- Option 3: Build and cache chart-index database.
  - Effort: high, requires invalidation.
  - Risk: stale cache complexity.
  - Impact: over-engineered for current scope.

Chosen: Option 2 (explicit override > changed-file parents > repo scan).

Decision B: Skip behavior when no charts discovered
- Option 1: Return FAIL with explicit instruction.
  - Risk: noisy in non-Helm repos; violates expected behavior pattern.
- Option 2: Return SKIP with explicit reason.
  - Risk: less strict visibility but less false failure.

Chosen: SKIP with explicit reasons (`SKIP all steps: no chart`) to maintain consistency with existing agents.

Decision C: Per-chart loop execution order
- Option 1: Process chart roots as discovered, run lint/template per chart immediately.
  - Effort: low.
  - Impact: direct and debuggable output.
- Option 2: Run all lint first across charts, then all template runs.
  - Effort: low but requires command batching.
  - Impact: less local context on first failure in per-chart diagnostics.

Chosen: sequential per chart per step for better locality and actionable logs.

## Code Quality Planning

### Organization and Module Structure
- Keep `helm-agent.sh` under ~200 domain lines; avoid duplicating shared lib patterns.
- Suggested function boundaries:
  - `usage()`
  - `collect_chart_targets()`
  - `run_lint_one_chart()` and `run_template_one_chart()`
  - `run_lint()` and `run_template()` (iterate chart list)
  - `main()`
- Keep CHART_DIR resolution and CHANGED_FILES parsing in dedicated helpers to isolate complexity.

### DRY Opportunities
- Reuse `step`, `print_result`, `fmt_elapsed`, `setup_outdir`, `setup_lock`, `print_overall`, `need`, and `should_continue` from shared library.
- Centralize path normalization and dedupe behavior in one helper so both lint/template share the same chart list.
- Centralize skip messaging constants for consistent user-facing wording.

### Explicit Error Handling (all edge cases)
- Missing required tool (`helm`) => `need helm` exits 2.
- `--help` path: must execute without `helm`.
- Unknown command => usage + exit 2.
- Empty `CHART_DIR` or non-existent path => `SKIP` or `FAIL` consistently; plan says choose `SKIP` when no discoverable chart root.
- `CHANGED_FILES` set with no matching chart-related files => skip decision with reason.
- Duplicate CHANGED file roots => dedupe once before running.
- `helm lint` failure: emit truncated diagnostics + fail hint; preserve full log.
- `helm template` success: discard rendered output; still emit PASS from per-chart step.
- `helm template` failure: emit failure context + fix hint.
- Chart directory with unreadable files/permission issues: FAIL with command output preserved for troubleshooting.
- `KEEP_DIR=1` or failing exit: logs retained; success defaults to cleanup.

### Technical Debt Prevention
- Avoid clever shell parsing or regex-heavy path logic that breaks on edge filenames.
- Keep `CHANGED_FILES` semantics explicit and deterministic.
- Keep fix-hints stable, exact, and short.
- Use small helper for chart dedupe to avoid divergence as helm, kube, docker agents are added.

### Engineering Level
Aim for “engineered enough”: deterministic, explicit branch handling, no premature generalization into a full filesystem graph resolver abstraction.

## Test Strategy

### Coverage Model
- Unit-like validation (functional, via scripted fixtures): chart resolver outcomes for
  - explicit `CHART_DIR`
  - CHANGED_FILES-derived chart(s)
  - repo-wide fallback discovery
  - no chart cases
- Integration behavior:
  - `tests/run-scenarios.sh helm-agent` clean/issues
  - `shellcheck --severity=warning`
- End-to-end acceptance:
  - `ralph validate work` and all impacted tests.

### Edge Cases + Corresponding Tests
1. Clean chart passes (`EXPECT_EXIT=0`, `RUN_ARGS=all`).
2. Broken chart fails (`EXPECT_EXIT=1`).
3. No chart in fixture -> skip all with explicit reason.
4. `CHART_DIR` explicitly set to valid chart path.
5. `CHART_DIR` set to invalid path.
6. `CHANGED_FILES` points at `.yaml/.yml/.tpl/Chart.yaml/values.yaml` in one chart -> scoped to that chart.
7. `CHANGED_FILES` contains only non-chart files -> skip (or skip-all behavior).
8. `CHANGED_FILES` references files across multiple charts -> both charts processed.
9. Duplicate CHANGED file entries -> single invocation per chart.

### Failure Modes and Error Paths
- `helm` missing.
- `helm lint` returns non-zero with syntax/templating errors.
- `helm template` fails due unresolved values.
- `CHART_DIR` resolution yields path without `Chart.yaml`.
- `FAIL_FAST=1` stops after first failing chart/step.

### Assertion Strength
- Assert exact exit codes (`EXPECT_EXIT`).
- Assert presence of: `Step:`, `Result: PASS|FAIL|SKIP`, `Fix:` for FAIL.
- Assert SKIP reasons are explicit and stable.
- Verify `Overall:` and `Logs:` appear.
- Confirm `All` command executes both steps unless toggles are disabled.

## Performance Considerations

- No DB access; all work is local file system + subprocess calls.
- Dominant costs are `find` and Helm CLI execution; avoid repeated discovery by resolving chart list once.
- Memory usage is bounded to per-step log strings and small path lists.
- No long-lived caches needed in first pass; optional future cache could track `Chart.yaml` paths for unchanged repos.
- Use `find` only when CHANGED_FILES is absent to avoid extra scans.

## Decision Log

- Decision 1: Make `CHANGED_FILES` branch subordinate to explicit `CHART_DIR` and fallback scan.
  - Options considered: (a) ignore `CHANGED_FILES` entirely for discovery; (b) always apply `CHANGED_FILES` first; (c) explicit override priority chain.
  - Chosen: explicit override first, then scoped discovery, then global discovery.
  - Impact: matches FR and avoids surprising runs when the user intentionally sets `CHART_DIR`.

- Decision 2: Keep `CHART_DIR`/no-chart conditions as `SKIP` rather than `FAIL`.
  - Options considered: fail hard on missing charts vs skip with context.
  - Chosen: SKIP with explicit reason (consistent with non-applicable tool behavior in existing agents).
  - Impact: less brittle CI when running in mixed-repo environments.

- Decision 3: Use per-chart sequential execution with failure aggregation.
  - Options considered: aggregate chart list and batch command once vs iterative per-chart loops.
  - Chosen: iterative per-chart with deterministic logs.
  - Impact: clearer logs and easier pinpointing when multiple charts exist.

## Edge Cases and Risks

- `CHART_DIR` explicitly set and points to a non-existent or non-chart directory.
- `CHANGED_FILES` contains entries with spaces (shared convention assumes whitespace-split; document and test skip/acceptance).
- `CHANGED_FILES` contains duplicates and nested chart files; ensure unique chart roots.
- Multiple charts affected in one run; ensure all are processed and aggregate outcome is correct.
- No `Chart.yaml` under cwd; both commands should skip with explicit reason and clean exit.
- Chart templates with invalid but non-fatal-looking syntax in helm lint mode; ensure template command still catches render failures.
- `helm` missing in environment when scenarios declare it absent; should skip via `REQUIRED_TOOLS` rather than script execution failures.
- Optional `FAIL_FAST` should stop orchestration after first failed step.
- Long-running charts should not fill stdout in memory; logs written to disk.
- `TEMPLATE` command output must be discarded (except log) to avoid large-memory and huge fixture logs.
- `install.sh` remote install path integrity and SKILL snippet auto-patching for `CHART_DIR` path hints.
