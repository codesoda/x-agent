## Architecture Review

### System Design and Boundaries

The go-agent should follow the existing architecture layering:

- `skills/go-agent/scripts/go-agent.sh`: domain-specific command runner only.
- `lib/x-agent-common.sh`: shared runtime helpers (outdir lifecycle, locking, formatting utilities, fail-fast flow, tool check).
- `tests/go-agent/*`: fixture contracts validated by `tests/run-scenarios.sh`.
- `install.sh` + `README.md`: packaging and discoverability boundary.

Primary coupling points:

- Script-to-lib path coupling via `SCRIPT_DIR/../../../lib`.
- External tool contracts to `go`, `gofmt`, `go vet`, and optionally `staticcheck`.
- Fixture contract coupling to `tests/run-scenarios.sh` (`scenario.env` fields and `EXPECT_EXIT`).

### Dependency Graph and Coupling Risks

- Mandatory runtime path: command script → source helper lib → invoke domain steps.
- High-coupling risk: all helpers are assumed to exist in `x-agent-common.sh`; any API change there affects all agents.
- Command coupling risk: `run_scenarios` can execute `go-agent` by path from `AGENT_SCRIPT`; install path integrity must hold.
- Data coupling risk: lockfile filename string in `setup_lock` is based on agent name, so using a wrong name breaks lock isolation.

### Data Flow and Bottlenecks

Flow:

1. Parse args, handle help/fail-fast.
2. Validate required tool (`go`), initialize outdir + lock.
3. Resolve `CHANGED_FILES` → package scope list.
4. Run selected steps in `all` with `should_continue`.
5. Write complete step logs to `$OUTDIR`, print truncated output and status.
6. Return PASS/FAIL summary.

Potential bottlenecks:

- `go vet`/`go test`/`go list` are the dominant costs; all else is O(n) path preprocessing.
- Staticcheck path is optional, so if installed it adds another full traversal.

### Security Architecture

- No new service/auth boundaries; only local filesystem and shell tool invocation.
- Inputs are env+args; command construction should avoid indirect eval-like behavior.
- Existing shell risk pattern (trusted project-local files and variables) remains; maintain existing conventions.
- Lock files are advisory; avoid predictable collisions outside agent namespace.

### Architectural Options and Tradeoffs

1) CHANGED_FILES package scoping strategy
- Option A: Direct directory derivation (`dirname` on changed `.go` files).
  - Effort: low, deterministic.
  - Risk: may include non-package directories; tool calls may produce extra work.
  - Impact: simple and fast.
- Option B: Resolve each changed directory with `go list` validation.
  - Effort: medium.
  - Risk: introduces extra command invocations.
  - Impact: more precise package targeting.
- Option C: Ignore CHANGED_FILES and always run on `./...`.
  - Effort: lowest.
  - Risk: slow and noisy; violates FR-3 requirement.

Recommendation: Option A with minimal `go list`-style fallback where needed. Keeps runtime low and satisfies FR scope behavior.

2) Fail hint construction
- Option A: Generic hints only.
  - Effort: low.
  - Risk: misses required `TestFoo` extraction detail.
- Option B: Parse `go test` output for named fails and join names in hint.
  - Effort: moderate string parsing.
  - Risk: brittle output variations; manageable with grep/sed patterns.
  - Impact: higher value diagnostics.

Recommendation: Option B (required by FR-12). Keep fallback generic if parsing fails.

3) CI format behavior
- Option A: Respect only user `FMT_MODE`.
  - Risk: conflicts with FR-10.
- Option B: `if CI` force check regardless of env.
  - Effort: minimal.
  - Impact: deterministic behavior and CI parity.

Recommendation: Option B.

### Implications for Future Specs

- This agent cements go-scoping helper expectations before US-05..10 start.
- If CHANGED_FILES resolution grows complex (e.g., vendor/module roots), consider adding a reusable helper in shared lib in a later spec.
- Ensure future agents that map changed files to target packages use similar scoped arg strategy for consistency.

## Code Quality Planning

### Module Structure and Organization

- Keep `go-agent.sh` under ~200 lines of domain logic by using shared lib helpers.
- Separate concerns by function:
  - env/usage
  - scope resolution
  - one `run_<step>` per command
  - `main()` dispatcher

### DRY and Duplication

- Reuse common output helpers (`step`, `fmt_elapsed`, `print_overall`, `print_result`, `setup_*`, `need`, `should_continue`).
- Avoid duplicating lock/setup/cleanup blocks.
- Build a single scoped target helper used by `fmt`, `vet`, `staticcheck`, and `test`.
- Shared log names by step: `<step>.log`.

### Explicit Error Handling (all edge cases)

- Missing required `go`: fail fast via `need go` with exit `2`.
- Missing optional `staticcheck`: report `SKIP` and continue.
- Missing `.go` files and/or no valid scoped dirs:
  - `fmt`: fallback to `.` in local/full mode.
  - vet/test/staticcheck: fallback to `./...` for safety.
- Command failures:
  - emit `Result: FAIL` and `Fix:` line with actionable command.
  - preserve full output under `$OUTDIR`.
- Empty fixtures/parse failures:
  - treat as non-fatal unless command output indicates actionable error.
- `help` invocation should bypass project/tool validation.

### Technical Debt Prevention

- Keep argument forwarding minimal and explicit to avoid shell injection and accidental argument loss.
- Preserve output contract (`Step`, `Result`, optional `Fix`, `Time`, `Overall`, `Logs`) and exit semantics.
- Use stable helper calls instead of hand-rolled traps/lock logic.
- Keep CHANGED_FILES parsing intentionally simple and documented to match existing repo conventions unless future spec explicitly lifts the space-in-filename limitation.

### Engineering Level

- Implement the smallest robust solution that meets FRs, avoid over-generalizing scoping beyond required `.go` behavior.
- Prefer readable Bash 3.2-compatible logic over compact but opaque string pipelines.

## Test Strategy

### Unit/Function-level

Because this repo is shell-driven, tests are primarily scenario-based, but internal behavior should still be validated by execution behavior:

- Run help path with no tools/project context to assert pre-tool usage handling.
- Simulate `CHANGED_FILES` with known `.go` file lists and verify scope output (when non-empty).
- Ensure `FMT_MODE=check` and `FMT_MODE=fix` are honored and that CI mode forces check.

### Integration (Scenario Tests)

- `tests/run-scenarios.sh go-agent` (must report no failures):
  - clean expects PASS
  - issues expects FAIL
- `tests/run-scenarios.sh shellcheck` (indirectly covered by full run) after script creation.
- `shellcheck --severity=warning skills/go-agent/scripts/go-agent.sh`.
- Manual execution inside fixture dirs:
  - `FMT_MODE=check` on clean and issue scenarios.
  - `FMT_MODE=fix` on issue scenario to validate rewrite and non-failing return (where fixture is only formatting issue).
  - `RUN_STATICCHECK=0` to verify command-level toggle behavior.

### Failure Modes and Edge Cases in Tests

- `go` missing in environment: scenario marked SKIP via REQUIRED_TOOLS in fixture.
- `staticcheck` missing: step must return SKIP and `all` continues to remaining steps.
- `CHANGED_FILES` with only non-`.go` entries: should degrade to unscoped package-level behavior.
- `RUN_*` toggles set to `0` skip intended steps.
- `--fail-fast=1` in all mode halts after first failure, and `overall_ok` is returned correctly.

### Assertion Strength

- Validate exit codes exactly (`EXPECT_EXIT`) through scenario harness.
- Validate each step’s status string (`Result: PASS|FAIL|SKIP`) and presence of `Fix:` on failures.
- Validate failure hints contain either actionable test names or explicit staticcheck skip wording.
- Validate logs are emitted and cleanup semantics follow `KEEP_DIR` and exit code.

## Performance Considerations

### Query/Execution Patterns

- Avoid running `go list ./...` multiple times in a single invocation when CHANGED_FILES is empty.
- Reuse a single resolved scope list for vet/staticcheck/test; do not recompute per step.
- Do not re-run `gofmt` on entire tree when scoped directories are known.

### Memory/CPU

- Keep process output on disk in per-step logs to avoid unbounded memory use.
- Use lightweight text scans (`grep`/`sed`) only after commands fail.
- Avoid storing huge package lists in command expansions where unnecessary.

### Caching Opportunities

- Cache scoped directory list once (environment variable/array) and reuse across steps.
- Reuse the same scope when command arguments are unchanged.

### Slow Paths

- `go test` and `go vet` are inherently expensive; scope reduction via CHANGED_FILES is the main optimization.
- staticcheck adds additional CPU; skip when unavailable.

## Decision Log

### Decision 1: Script source path strategy
- Alternatives:
  - A) Use `SCRIPT_DIR/../../../lib/x-agent-common.sh` (same as existing agents).
  - B) Add lib path resolver utility in common layer.
  - C) Use env override for lib path.
- Chosen: A.
- Why: explicit requirement alignment and lowest churn; predictable for this spec.
- Implications: keep in sync with existing install/local path assumptions.

### Decision 2: Scope target derivation for changed files
- Alternatives:
  - A) Use file-directory projection only.
  - B) Resolve each directory through `go list`.
  - C) Ignore CHANGED_FILES.
- Chosen: A with safe fallback to full run.
- Why: meets FR while keeping runtime low and implementation compact.
- Implications: behavior remains intuitive and fast; future helper extraction can refine precision.

### Decision 3: Test failure hint extraction
- Alternatives:
  - A) Generic failure message.
  - B) Parse `--- FAIL:` and include test list.
  - C) Parse package-wide summary lines.
- Chosen: B.
- Why: directly satisfies FR-12 and improves remediation speed.
- Implications: depends on go test output format; add fallback generic hint if parse misses.

### Decision 4: Install/README updates
- Alternatives:
  - A) Delay and only rely on hidden agent selection.
  - B) Register in `install.sh` and docs immediately.
- Chosen: B.
- Why: aligns with project standards and user discoverability requirements.
- Implications: no discoverability gap when agent is added.

## Edge Cases and Risks

- `CHANGED_FILES` containing spaces: consistent with current project convention, this splits on spaces and may mis-handle such paths.
- `gofmt -l` returns non-zero only on parse errors; must use output emptiness as pass/fail in check mode.
- Staticcheck skip path must not abort agent execution.
- go test can return fail with output in stderr only; capture both streams consistently.
- `--fail-fast` with `all` and scoped steps must preserve `overall_ok` and return code semantics.
- `go test` failures can include package-level `FAIL` banners without any `--- FAIL:` lines; still provide generic fix hint.
- `all` mode under `KEEP_DIR=0` must preserve logs only on failure.
- `FMT_MODE=auto` plus explicit `FMT_MODE=fix/check` in environment should have deterministic priority.
