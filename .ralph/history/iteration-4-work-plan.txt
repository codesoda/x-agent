## Architecture Review

### System Boundaries

The runtime is split into three layers.

1) Shared runtime layer in `lib/x-agent-common.sh` holds boilerplate that is structurally identical across agents: output helpers, environment defaults, tool checks, output-dir lifecycle, lock lifecycle, and final summary formatting.

2) Domain runner layer in `skills/<agent>/scripts/<agent>-agent.sh` holds project-specific logic only: command dispatch, tool arguments, package/tool-specific step functions, and domain-specific error messaging.

3) Test/packaging layer in `tests/run-scenarios.sh` and `install.sh` exercises contract and distribution.

`tests/run-scenarios.sh` is a strict integration boundary because all behavior claims are verified by the fixture contract, not by unit tests.

### Dependency Graph and Coupling

- Hard dependency: each edited script depends on `lib/x-agent-common.sh` at runtime.
- Direct coupling path: `SCRIPT_DIR -> ../../../lib -> lib/x-agent-common.sh` (fragile if scripts are relocated).
- Shared optional dependencies: `flock`, `perl`, `bash`, and command-line tools per agent.
- Packaging coupling: `install.sh` must ensure `lib/` is installed in both local and remote modes.

The explicit coupling is acceptable for this phase but should be tracked because path-coupling is the highest single fragility point.

### Data Flow Patterns and Bottlenecks

1) Input/env processing: args and knobs are resolved first (`RUN_*`, `KEEP_DIR`, `FAIL_FAST`, `CHANGED_FILES`, `MAX_LINES`, `TMPDIR_ROOT`).
2) Setup phase: create `OUTDIR`, install exit trap via `setup_outdir`, acquire lock via `setup_lock`.
3) Execution phase: run ordered steps, each writing command output to `$OUTDIR/<step>.log` and printing truncated diagnostics + elapsed time.
4) Aggregation: update `overall_ok`, optionally short-circuit with `should_continue`.
5) Summary phase: call `print_overall "$overall_ok"`, then return exit code.

Primary bottlenecks remain external tool execution (`cargo`, `node`, `terraform`), not shell orchestration.

### Security Architecture

- Input vectors are local shell arguments and environment variables.
- Risk surface is command execution from user workspace; current pattern is command-injection-prone only if shell variables are interpolated unsafely. Existing scripts already pass this through unquoted args in places requiring shell parsing, so security posture is mostly inherited and must remain unchanged.
- Lock files are local and only advisory. Existing fallback to perl ensures compatibility but does not provide cryptographic guarantees.
- `install.sh` path rewriting and symlink/copy behavior should remain unchanged to avoid path injection from remote fetches; this phase only inherits those mechanics.

### Architectural Decision Options

1) Option A: Keep only hardcoded source path `../../../lib` and no abstraction.
Why: aligns with FR-2 requirement text and current install layout. Lowest code churn.
Cost/Risk: tight coupling to script location and difficult future portability.
Impact: fastest path, minimal change footprint.

2) Option B: Introduce `resolve_lib_dir()` in `lib/x-agent-common.sh`.
Why: centralizes path logic and reduces relocation risk.
Cost/Risk: deviates from explicit required snippet, may require additional behavior changes and tests now.
Impact: better future flexibility, higher refactor risk for this phase.

3) Option C: Add per-agent `LIB_DIR` override env fallback (e.g., `RA_LAUNCHER_LIB_DIR`).
Why: allows tests or installs to override without path math in scripts.
Cost/Risk: extra configuration surface and allowed-tools complexity.
Impact: flexibility increase, but overkill for this narrow spec.

Recommended: Option A for this phase because it matches explicit spec requirement and previous extraction assumptions, with a follow-up ticket for optional Option B once all agents are stable.

### Implications for Future Specs

- All future agent work should preserve helper boundary and call pattern for consistency.
- Future specs can de-duplicate path coupling by upgrading the shared lib to Option B after this phase.
- Any new agent should be validated against the same `run-scenarios` contract before introducing more helpers in the shared layer.

## Code Quality Planning

### Module Organization

The goal is a clean separation between shared runtime helpers and domain step implementations.

1. Keep `lib/x-agent-common.sh` as a pure helper module with no side effects on source.
2. Keep agent scripts minimal and domain-centric: parser, run_* functions, and dispatch only.
3. Keep all `CHANGE` behaviors (`CHANGED_FILES`) in domain scripts because semantics differ by tool.

### DRY Opportunities

Most duplication is exactly the boilerplate block, not business logic.

- Remove repeated block variables and helper functions from all three scripts.
- Reuse `setup_outdir` and `setup_lock` exactly once in each script.
- Keep helper calls like `print_overall` only where identical to avoid introducing non-equivalent wrappers.

### Error Handling Patterns to Preserve

- `need` must continue to exit 2 for missing required tooling.
- `cleanup` semantics must remain: preserve `$OUTDIR` on failure or `KEEP_DIR=1`, remove on success otherwise.
- Help path (`help`, `--help`, `-h`) must still run without project preconditions.
- Lock wait message must still print when lock contention exists.
- `overall_ok` must preserve step-level fail state.

### Explicit Edge Cases to Handle

- If script receives unknown command: exit 2 and usage message.
- If required tool missing: immediate 2 with same message shape.
- If lock mechanism unavailable and neither flock nor perl present: warning with continuation.
- If running under CI (`CI=true|1`): `MAX_LINES` unlimited and check-only behavior unaffected.
- If `RUN_*` is set to 0: skip step exactly as before.
- If `FAIL_FAST=1`: stop after first failure in multi-step mode.
- If `KEEP_DIR=1`: trap must preserve path and exit still returns original status.

### Technical Debt Prevention

- Keep source block order strict and canonical to minimize later merges.
- Avoid in-script partial duplication of print logic to reduce drift from shared library contract.
- Do not alter command argument handling for existing agent features; only move boilerplate.
- Preserve line-level output where feasible, because this is a behavioral contract and tested by snapshots.

### Engineering Level

Use minimal-variance refactoring. Move boilerplate only, preserve command ordering and output contracts, and avoid introducing generic abstractions beyond what this spec requires.

## Performance Considerations

### N+1 and heavy access patterns

- Avoid adding repeated `cargo metadata` calls; keep current one-time/conditional invocation patterns.
- Keep `CHANGED_FILES` scanning and log directory creation per-agent unchanged to avoid additional process overhead.
- Avoid introducing extra `cat`/`grep` passes over large files in step routines.

### Memory usage

- Shared helper calls are lightweight and do not introduce persistent large data structures.
- `terraform` and `cargo` outputs remain on disk in log files as before; no in-memory buffering explosion risk.

### Caching opportunities

- Existing cargo helper already caches metadata in `_WS_PACKAGES` and similar variables. Preserve this as-is.
- Avoid clearing/transforming shared cached variables from `lib/x-agent-common.sh` so agent-level caches remain local.

### Slow paths and optimization points

- Lock contention path is intentionally blocking by design to preserve correctness under concurrent runs.
- CLI argument parsing and env resolution are constant time.
- Biggest runtime cost is external commands; no additional slow shell loops should be added.

## Decision Log

### Decision: Use shared setup helpers or retain per-agent setup code
- Decision: Use `setup_outdir` and `setup_lock` from `lib/x-agent-common.sh` across all three scripts.
- Options Considered: retain each agent's existing setup logic; add partially shared wrappers while preserving original helpers; full shared helper adoption.
- Chosen Approach: Full adoption of shared helpers because this directly validates FR-1 and reduces drift.
- Implications: simpler scripts, reduced duplicated lifecycle logic, but requires stronger parity checks for output and lock timing.

### Decision: Keep source path pattern as mandated snippet
- Decision: Use `SCRIPT_DIR` + `LIB_DIR="${SCRIPT_DIR}/../../../lib"` exactly as documented.
- Options Considered: absolute path detection, fallback env override, `resolve_lib_dir()`.
- Chosen Approach: mandated hardcoded relative path for this phase.
- Implications: fast and compliant with the spec contract; leaves explicit coupling to script layout.

### Decision: Replace common final summary block with shared printer
- Decision: Replace only equivalent final summary output with `print_overall`.
- Options Considered: keep all final blocks inline; partial extraction with `print_result` only; full shared finalization.
- Chosen Approach: use shared `print_overall` where behavior matches exactly.
- Implications: less output duplication and consistent overall output, with explicit parity diff to guard accidental formatting drift.

## Edge Cases and Risks

- Output parity drift is possible in timing and trailing blank-line handling. Mitigation: normalized-diff comparison before final acceptance.
- Lock path behavior can drift if setup order changes. Mitigation: run flock test fixture and compare message sequence.
- `help` command must remain fast and independent of project root validation. Mitigation: keep early usage handling in each main function.
- Optional tool absence can become false failures. Mitigation: scenario-required tools gating should remain unchanged and no new unconditional checks should be introduced.
- Sourcing shared library without side effects must be preserved. Mitigation: no extra output in source phase; no additional initialization on source.
- Path coupling risk if repo structure changes later. Mitigation: log as known coupling in exploration and architecture notes, then plan optional `resolve_lib_dir` follow-up.
