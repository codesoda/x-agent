## Patterns & Conventions
- Scripts use Bash 3.2-compatible syntax and start with `set -euo pipefail`.
- `--help` / `help` / `-h` is handled before any tool checks so help works in environments without validators.
- Required dependencies fail with exit code `2` using `need` or custom logic that prints explicit install guidance.
- Output contract remains strict: each step prints `Step`, `Result`, optional `Fix` for failures, `Time`, and final `Overall` and `Logs`.
- `CHANGED_FILES` is optional and used only where meaningful; if non-empty, logic narrows execution targets.
- Scenario contract (`SCENARIO_NAME`, `AGENT_SCRIPT`, `RUN_ARGS`, `EXPECT_EXIT`, optional `REQUIRED_TOOLS`) is standardized in `tests/run-scenarios.sh`.
- `shellcheck --severity=warning` runs across `skills/*`, `lib/*`, and test shell scripts before scenario execution.

## Database Schema
- No runtime DB/schema is used by the agent runtime or test harness.
- State for loop execution lives in `.ralph` files (`ralph-state.json`, logs, and task iteration files).

## Test Infrastructure
- Core harness: `tests/run-scenarios.sh` discovers `tests/*/scenario.env`.
- A fixture can set any env values consumed by the agent script, including `CHANGED_FILES` and `KUBE_SCHEMAS_DIR`.
- Exit semantics: `EXPECT_EXIT=0` for pass, `1` for fail, `2` for config/dependency failure paths.
- Tool availability is injected via `REQUIRED_TOOLS` in each `scenario.env`.
- Existing test pattern: each new agent ships at least `clean/` and `issues/`, plus optional scoped/skip fixtures when FRs demand.

## Architecture & Data Flow
- Input environment and CLI command enter the agent script.
- Script resolves validator and candidate files, optionally filtered by `CHANGED_FILES`.
- Validation runs once per step with logs written to `OUTDIR`.
- Structured contract prints per-step and overall result, including actionable `Fix` lines.
- Errors and skips are returned via exit codes while preserving logs for debugging.

- Installer path: `install.sh` pulls from `SKILLS` list; for `kube-agent`, this list and optional checks must be extended.
- Documentation path: `README.md` must stay aligned to skill script names and command set (`validate`, `all`, help variants).
