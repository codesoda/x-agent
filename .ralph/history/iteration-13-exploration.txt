# x-agent Exploration Notes

## Key Files & Roles

- `lib/x-agent-common.sh`: shared runtime library for agent scripts. Provides environment defaults, output helpers, cleanup/trap setup, lock handling, and shared result formatting utilities.
- `skills/cargo-agent/scripts/cargo-agent.sh`: refactored agent using the shared library; pattern for complex scoped execution and structured output.
- `skills/go-agent/scripts/go-agent.sh`: refactored single-step and multi-step domain runner with CHANGED_FILES mapping and CI-aware format behavior.
- `skills/npm-agent/scripts/npm-agent.sh`: refactored agent that demonstrates formatter/linter fallbacks and scoped execution over changed files.
- `skills/terra-agent/scripts/terra-agent.sh`: refactored agent with optional tooling (tflint), non-mutating checks, and lock-aware sequencing.
- `tests/run-scenarios.sh`: central test orchestrator for shellcheck plus fixture execution.
- `tests/<agent>/clean/` and `tests/<agent>/issues/`: per-agent scenario fixtures; each includes `scenario.env` with `AGENT_SCRIPT`, `RUN_ARGS`, `EXPECT_EXIT`, and `REQUIRED_TOOLS`.
- `install.sh`: installer entrypoint; currently installs `bash-agent`, `cargo-agent`, `go-agent`, `npm-agent`, `py-agent`, and `terra-agent`, and patches shared library for local/remote install modes.
- `README.md`: human-facing agent catalog, usage examples, and quick-start guidance.
- `.github/workflows/ci.yml`: CI currently runs scenario slices for bash, cargo, go, npm, py, and terraform agents.
- `.ralph/specs/05-gha-agent.md`: current specification for the new gha-agent implementation.
- `docs/agents/add-x-agent.md`: reference for required files, trigger patterns, and required helpers for each new agent.
- `docs/agents/definition-of-done.md`: final acceptance checks for any new agent.
- `.ralph/task.md`: task-level scope and spec sequence for eight remaining agents.
- `.ralph/automation-opportunities.md`: pending candidates for automatable process improvements.
- `.ralph/lessons.jsonl`: historical manual lessons and decision points.

## Patterns & Conventions

- Bash scripts use Bash 3.2-compatible syntax and start with `#!/usr/bin/env bash` plus `set -euo pipefail`.
- Agents print canonical Step/Result/Time output and end with `Overall` and `Logs`.
- Every `Result: FAIL` includes a `Fix:` line; failures preserve full logs and print truncated context locally.
- Universal knobs are expected in all agents: `RUN_<STEP>`, `FAIL_FAST`, `MAX_LINES`, `KEEP_DIR`, `TMPDIR_ROOT`, and `CHANGED_FILES`.
- CI behavior for scripts is controlled by `CI=true|1`; line truncation is disabled and check-only mode is preferred for mutating steps.
- Scripts should parse help flags before running `need()` checks so `--help` works without project context.
- `--fail-fast` is supported in `all` mode via shared `should_continue()` guard.
- Workflow-level locking is centralized in shared helpers using `flock` with Perl fallback.
- Missing optional tooling is expected to produce `SKIP` with explicit reason; required tooling exits 2 through `need()`.
- `tests/run-scenarios.sh` skips scenarios whose `REQUIRED_TOOLS` are unavailable.

## Database Schema

No database is used.

## Test Infrastructure

- Scenario discovery happens via `find tests -name scenario.env`, including `clean` and `issues` fixture pairs per agent.
- `tests/run-scenarios.sh` can run all tests, a named fragment filter, `--list`, and shellcheck-only mode.
- Shellcheck is run with `--severity=warning` and is required to pass for all `*.sh` files under `skills`, `tests`, `lib`, and root installer scripts.
- Fixture assertions are exit-based using `EXPECT_EXIT`; mismatches show log slices from the failing fixture.
- Scenario fixtures declare `REQUIRED_TOOLS` and are skipped automatically when missing.

## Architecture & Data Flow

- Invocation enters through a script under `skills/<agent>/scripts`.
- Shared library is sourced early after env defaults and before agent-specific logic.
- On command execution: parse CLI/help/flags, validate required tools, initialize output directory and lock, resolve scope from `CHANGED_FILES`, execute selected steps, and write each step log under `$OUTDIR`.
- Steps run external tooling with scoped targets when available, write logs to `$OUTDIR/<step>.log`, and print status summaries.
- Final status is aggregated into `overall_ok` and rendered via `print_overall`.
- `install.sh` and `README.md` are the registration/discovery boundary for new agents.
- For gha-agent, the data path is `.github/workflows/*.yml|*.yaml` from the project root; no existing gha-agent implementation is currently present in repo.
